# Part I — Scope, strategy & phased roadmap

## Scope & thesis

**Thesis.** MessageFoundry already ships broad, high-quality coverage on the single-node SQLite path. The job of this plan is not to re-prove that surface but to **close the gaps around it** — the places where a regression would be *silent* (no failing test) and *material* (PHI, data loss, or a security bypass). We treat the existing ~397 tests, the **WIN2025 host-acceptance framework**, and the **load/failover/reconcile harness** (`harness/load/`) as load-bearing assets and **layer on** them: every phase references the suite or harness it extends and adds only the missing assertion, backend leg, live run, or detector.

The 128 gaps decompose into six coverage **dimensions**, and the whole plan is organized so each dimension is driven to closure across all three store backends (SQLite / SQL Server / PostgreSQL) and the engine-shard vs database-shard axes.

### The six dimensions
1. **Functional** — the feature does what its ADR/AC says (happy path + error/edge branches, dead-letter routing, at-least-once).
2. **Security** — fail-closed guards actually refuse (SSRF/redirect/cleartext-hop/mTLS/allowlist/injection), asserted by a *negative* test, not by construction.
3. **Performance** — throughput/latency/scaling claims have a repeatable gate (or an explicit "measured off-repo / harness-only" disposition).
4. **HA / failover** — behavior under leader change, crash-restart, and multi-node, proven on a real server DB, not a fake pool.
5. **PHI** — no message body / element value / needle reaches a log, stdout, exception, audit row, or report; at-rest data is encrypted on every backend.
6. **Cross-backend** — SQLite-proven behavior is executed (not just structurally asserted) against SQL Server and PostgreSQL, including the deliberate per-backend divergences (SS response-column cipher pass, PG NUL rejection, `body_ref` inert on servers).

### Relationship to what already exists (do NOT duplicate)
- **Existing pytest suites** own the single-node SQLite functional/security proofs. New work adds the *missing branch or backend leg*, keyed to the same test module where possible.
- **WIN2025 host-acceptance matrix** owns manual/host checks (NSSM install, ACLs, AD-lab, reboot/crash). P6 converts what can be automated into CI legs and leaves the genuinely manual items attributed to the matrix — it does not re-invent them.
- **`harness/load/` (connscale / failover / reconcile / falsifier discipline)** owns throughput, 1,500-connection, SIGKILL-under-load, and GO/NO-GO verdicts. P4/P5 add *in-CI gates or reference-config baselines* around the harness; they do not re-implement the rig.
- **ADR falsifier record (0069/0098/0107)** already settled that group-commit, inline-fusion, and the txn lever are dead. P7 marks those **dormant/withdrawn** with guard tests; it explicitly does **not** commission new perf work on them.

### What NOT to re-test (already covered — leave alone)
- Single-node SQLite functional paths marked `coverage:covered` (e.g. MLLP framing/ACK build, FILE atomic write/path-confinement, DICOM codec + SCP/SCU classification, X12 codec/framing, FHIR SSRF grammar-gate, all `db_lookup` read-only enforcement, parsing peek/message-model/escape codec, RBAC role catalog/field-authz/hash-chain, auth TOTP/passkey/session core, the whole `/ui` CSRF/CSP/autoescape surface).
- Any `covered` row whose only residual is a `backend:partial` or `perf:partial` **note** — those are swept opportunistically inside P1/P5, not tracked as first-class gaps here.
- **Misfiled `covered` rows** flagged by the critic: **MLLP-23** (detached-JWS has no MLLP path — substantive coverage is HTTPFHIR-22; do not build an MLLP signing test) and **PIPE-17** (Direct S/MIME is a transports connector, already well covered). These are noted, not scheduled.
- Declined-by-design items (visual/template authoring, serial/ASTM, database-shard split L5) — recorded, never tested.

### PHI / data discipline (hard rules for every phase)
- **Synthetic HL7 only.** Use `messagefoundry generate` corpora and the anonymized harness corpus (ADR 0030). Never real PHI in a test, fixture, log, or CI artifact.
- **No bodies in reports.** This plan, its tables, and any generated CI output carry IDs, counts, dispositions, and SQLSTATEs only — never a message body, element value, or search needle.
- **PHI-no-log is a first-class assertion.** Where a gap is PHI-dimensioned (DICOM-23, INGEST-4, ANON-9, PARSE-14 exc_info, DEPLOY-6/11), the new test asserts absence from logs/stdout/exception/audit at DEBUG level — it does not merely exercise the happy path.
- **`dryrun`/`generate`/`--show-phi`** output is treated as body-bearing and never redirected to a committed file or CI log.

### Definition of done (per gap and per phase)
A gap is **done** when: (a) the missing dimension has an executable assertion (or, for design-only/dormant items, a written build-or-supersede decision + a guard test that the status holds); (b) for cross-backend gaps, the assertion **executes** on SQLite + SQL Server + PostgreSQL (gated legs acceptable, but run in CI, not structural-only); (c) new behavior passes `ruff`/`mypy --strict`/`pytest` (Qt tests `QT_QPA_PLATFORM=offscreen`); (d) no PHI appears in any artifact. A **phase** is done when all its gaps meet the bar and its exit criteria hold. The plan's north star — every coverage gap closed across all six dimensions and all three backends — is met when P0–P6 exit and P7's dispositions are recorded.
