## 3. Backlog #1 — verification verdict (SQL Server store concurrency safety)

"Backlog #1" = `../BACKLOG.md` §1 (review findings H-6/H-7/H-8/M-6, low-2 + low-3 store half) — the
named correctness/data-integrity defects on the SQL Server backend, **not** the staged-pipeline item.
Because SQL Server is now a production backend, these are production-critical; **all five are verified
FIXED** in `messagefoundry/store/sqlserver.py`:

| # | Defect (review id) | Verdict | Evidence on `main` |
|---|---|---|---|
| 1 | Audit hash-chain append race (H-7) | **FIXED** | `record_audit` serializes read-prev-then-insert under `self._audit_lock`; `_backfill_audit_chain` runs in `open()` before the store is returned. |
| 2 | Finalize deadlock / missing RCSI (H-8/M-6) | **FIXED** | RCSI enabled pre-pool (`_ensure_rcsi`); finalizer serialized by transaction-scoped `sp_getapplock` keyed `mefor:finalize:{mid}` — a named lock that touches no `queue`/`messages` rows, so no AB/BA inversion vs producers. |
| 3 | Inert pyodbc command-timeout (STORE-3) | **FIXED** | `_acquire` sets `raw.timeout = command_timeout` on the underlying `pyodbc.Connection` per-acquire; the prior `conn.timeout=` raised `AttributeError` and was swallowed. |
| 4 | Rollback hygiene in `_fetchall` / bootstrap (M-6, low-2/3) | **FIXED** | `_fetchall`/`_execute` roll back before re-raise, so a failed read can't return a mid-transaction connection to the pool. |
| 5 | Non-atomic `upsert_role` (concurrent seeding → PK violation) | **FIXED** | Single `MERGE roles WITH (HOLDLOCK)`; the range-lock prevents two seeders both inserting. |

**Verdict: #1 is RESOLVED.** The only residual is a **doc edit** — `../BACKLOG.md` §1 still labels the
backend "experimental / fix before promoting." Mark it DONE / move to shipped history (folds into the
Wave 0 doc sweep; not on the critical path).

## 4. Resolved owner decisions (Q1–Q7)

All seven open questions from `v0.1-PLAN` are resolved (the judgment calls were adversarially refuted).

**Q1 — Reference hardware:** *the deploying org provisions it.* The project ships harness +
methodology + a labeled **reference-config** baseline run on accessible infra (Docker SQL Server 2022 +
PostgreSQL 16 containers + SQLite on the owner's box / CI), plus per-org tuning guidance. **This takes
hardware procurement off the critical path entirely.**

**Q2 — Duplicate CVE PRs:** **no action** — already resolved. #271 merged (clears all three advisories
via clean `fastapi`/`cryptography` floors), #272 closed and branch deleted, `pip-audit` green, DEP-1 in
sync. Wave 0 CVE allocation = **0**.

**Q3 — Throughput SLO:** reframe Gate #3 into a **two-tier gate** (detail §6):
- **Conformance tier** — hardware-*independent*, a **hard non-overridable** release blocker: zero
  message loss, bounded drain, error rate < 0.1%, low ACK-on-receipt p99 (all already measured by the
  harness via `sent==engine_read`, `sink_received==engine_written`, `backlog→0`).
- **Performance tier** — the existing fanout numbers (**≥200 msg/s sustained, ACK p99 ≤50 ms, e2e p99
  ≤5 s**), **reclassified** as the floor the *named reference config* must clear, reported "as measured
  on config X" and flagged as a sanity floor far below hospital peak (not a market claim).
- **Miss policy:** a conformance-tier miss blocks the tag (no override); a performance-tier miss → fix
  and re-run **or** owner sign-off recording the measured value + architectural reason **in the same
  commit** — never a silent threshold lowering.
- Publish alongside it the verified note that **no trustworthy public per-node competitor benchmark
  exists** (cited Rhapsody/Mirth figures are refuted marketing; only IRIS publishes a rigorous
  host-bound number — the *form* MF emulates).

**Q4 — Gate #4 deployment doc:** **author `docs/DEPLOYMENT.md`** (the facts are real but scattered
across ~5 docs and incomplete; PHI.md §4's data-in-transit table is only 5 rows). Core = one
**channel × {bind default, TLS support, auth, egress gate}** matrix over **all ~13 channels** (count
X12 as its own connector; flag raw-TCP and plaintext FTP as the genuine no-TLS rows; state inter-node
coordination TLS = the shared store-DB connection's posture for both backends) + a short "before you
expose off-loopback" checklist. **S, docs-only, parallel** — the last item to mark Gate #4 acceptance.

**Q5 — Pre-exposure hardening (PR #269):** only **(c) is v0.1**.
- (a) cleartext-at-rest hardening → **0.2** (warning half near-done — prod+staging already warn; the
  remaining piece is a *breaking* `require_encryption` fail-closed flip).
- (b) egress deny-by-default → **0.2 globally** (a global flip would `WiringError` every un-enumerated
  outbound and refuse startup for real configs), **but** ship a v0.1 **XS** slice: a prod/staging

- (c) certificate-expiry AlertSink → **v0.1 (~S)** — deferral realizes a *new* exposure-specific

  Workstream E (consumes the WP-13a/13b cert config).

**Q6 — Console scope for v0.1:** ship **exactly two** items, as **one coupled PR** (polling-worker

## 6. Gate #3 detail (the long pole)

Now unblocked (Workstreams A and B both merged) and de-risked by the Q1/Q3 reframe (no hardware
procurement). Remaining work:

1. **Failover-load harness capability (net-new).** Today the active-passive failover coverage lives in
   `tests/test_cluster*`, not the load harness. Build a harness-side mid-load primary-kill measurement:
   recovery/promotion time + zero duplicate/loss + preserved per-lane ordering. This run is also the
   **first live proof** of the SQL Server on-promotion recovery path (`reclaims_inflight()==True` makes
   the engine skip the unconditional startup reset, so recovery rests on the on-promotion
   `reset_stale_inflight` — unit-covered but never exercised under a simulated partition under load).
2. **`in_pipeline` `/stats` gauge.** `stats()` is outbound-only today, so the harness's no-loss/drain
   check falls back to a `read`/`written`-stability heuristic that **cannot distinguish a stalled
   router/transform** from a true drain. Add a `COUNT(*) FROM queue WHERE status IN ('pending','inflight')`
   across **all** stages as an `in_pipeline` key on `stats()` (a **3-backend** change — `store.py` +
   `sqlserver.py` + `postgres.py`, since `stats()` is a Store-protocol method); change `await_drain` to
   require `in_pipeline == 0`. **Hard prerequisite of the published baseline runs.**
3. **Closed-loop max-rate profile.** The `governor._run_closed` mode already exists; only the profile
   TOML + a parse test are missing (small).
4. **Run + publish.** Execute fanout-baseline + closed-loop against **SQLite → PostgreSQL → SQL Server**
   plus the failover run, on accessible infra. Commit **metrics-only** artifacts to a tracked location
   (e.g. `docs/benchmarks/results/`; `out/load/` is git-ignored) — no message bodies, no control-ids;
   must pass the publish denylist guard.
5. **Docs.** Author `docs/benchmarks/TUNING-BASELINE.md` (lead with "how each deploying org validates on
   *their* hardware") and `docs/DEPLOY-SERVER-DB.md` (both are already cited by other files → broken
   links until written).
6. **Verdict.** Conformance tier must pass (hard); performance tier meets the reference-config floor
   **or** carries an owner-signed revised value in the same commit (§4 Q3).

## 9. Deferred items — first-draft build plans

### 9.1 Backlog #3 — Per-key (partition-key) ordering (0.2, **XL**)
Prerequisites are on `main` (FIFO-per-lane, SQL Server `seq IDENTITY`, `peek.field`). **Note (2026-06-18):
the Track B per-lane leasing this plan referenced was removed with the active-active drop — per-key
ordering now parallelizes *within* a single engine (single-leader active-passive), so the multi-node
lane-leasing prerequisite and step 4 below no longer apply.** Owner constraint: strict FIFO is THE model
— never UNORDERED.
0. **ADR 0018 (gating, no code):** `partition_key` per outbound connection (optional router override),
   a small allow-listed expression (start PID-3.1 MRN / PV1-19 visit / MSH-4 facility); **default unset
   = one lane = byte-identical**; lock the **A40 merge fallback = serialize on the base (key-less) lane**.
1. Schema: nullable `order_key` (default `''`) on `queue` across **all three backends in lockstep** +
   FIFO indexes; stamp at the router stage. With no key configured, claim is identical — **ship
   byte-identical first**.
2. Keyed claim: `AND order_key=?` in the lane filter; per-(lane,key) `_fifo_created_at` clamp (SQL Server
   lacks the SQLite helper today); bounded per-outbound fan-out worker draining each key strictly FIFO.
3. A40 cross-key serialization: classify merge/move triggers (A40/A18/A30/A34/A39 family) → force
   `order_key=''`; configurable trigger set; document the guarantee.
4. ~~Track B: extend `lane_leases` PK + ownership to include `order_key`.~~ **(N/A since 2026-06-18 — `lane_leases` / per-lane ownership was removed with the active-active drop.)**
5. Observability + docs (keyed lane depth/age, cross-key-serialization counter).

**Key hazard:** A40 patient-merge correctness — naive per-MRN parallelism reorders a merge spanning two
MRNs → silent PHI corruption; the serialization fallback must be conservative and default-on. Secondary:
unbounded key cardinality on a firehose (mandatory concurrency cap); three-backend lockstep.

### 9.2 Backlog #6 — IDE functional tests (0.2, **M**)
`ide/` has no runtime tests; CI only does `npm ci → tsc --noEmit → esbuild`.
1. Add `@vscode/test-electron` (+ `@vscode/test-cli`/mocha) as devDeps — **verify they exist + pin**, then lock.
2. Add a runner + suite that downloads a pinned headless VS Code, opens a fixture workspace, **activates
   the extension**, and asserts each contributed command registers + fires without throwing (start with
   `setup`/`promote`/`test bench`).
3. Add `npm test`; wire into the `ide` CI job after the build, under `xvfb` on Linux.
4. Add 2–3 fixtures.

**Key hazard:** headless VS Code in CI needs a virtual display + a network download of the pinned build;
pin the version and cache the download to avoid flakiness.

### 9.3 Backlog #7 — Inbound SOAP/REST HTTP-listener source (0.3, **L**)
Genuinely unbuilt — `rest.py`/`soap.py` are destination-only. Enabling contract (ADR 0004
payload-agnostic ingress) is merged; ADRs 0015/0016 are MEFOR-as-client and exclude an inbound listener;
**0017 is the next free ADR number.**
0. **ADR 0017 (gating, no code):** ratify the **respond-with-receipt** seam for the MVP — commit raw to
   the **ingress** stage → HTTP 2xx (the MLLP AA-on-ingress analog); decode/validation failure records
   `ERROR` before any ingress row → HTTP 4xx (NAK analog). Listener owned by `transports/`, **not** `api/`
   (one-way dependency, CLAUDE.md §4); off-loopback TLS from ADR 0002; inbound auth; SOAP XXE/DTD
   defenses reused from ADR 0015.
1. `transports/rest_in.py` `RestListenSource` owning an asyncio HTTP server (**prefer stdlib — no web
   framework in the engine**), cloned from `MLLPSource.start`: per-node bound socket, **not** leader-gated.
2. Widen the inbound reply contract (`InboundResponse(status, body, headers)` or keep `str|None` for MVP)
   — keep the MLLP/HL7 path byte-identical (additive only; regression-test hard).
3. Wiring + validation (`ConnectorType.REST` source; `RestListen()` factory; require TLS on non-loopback bind).
4. SOAP-IN phase 2 (envelope + Fault, ADR-0015 hardening gate).
5. *(Optional)* synchronous capture-and-block via the ADR-0013 response machinery.

**Key hazard:** the synchronous-response seam is an invariant collision (ACK-on-receipt + count-and-log
vs a caller that blocks for an answer); respond-with-receipt preserves the invariants cleanly. This is
the engine's first partner-facing inbound socket beyond MLLP — needs auth/TLS/request-size limits.

### 9.4 Load-harness follow-ups (0.2 / opportunistic, **M** bundle)
1. **Broaden the lock-free SQLite reads** beyond the three metrics reads already serialized. Recommended:
   a dedicated **read-only WAL connection** (`PRAGMA query_only=ON`) for the list/get reads, exploiting
   WAL one-writer/many-readers so dashboards never contend on the write lock (accepted cost: a slightly
   stale snapshot). Fallback: serialize each read on `self._lock` (uniform but queues browses behind
   deliveries). SQLite-store-specific (PG/SQL Server use pools).
2. **`in_pipeline` drain gauge** — folded into Gate #3 (WS-D) for credibility; 3-backend change.
3. **Single-source the publish-guard token list** — fold the additional real-estate tokens the harness
   denylist test enumerates into `scripts/publish/scan_forbidden.py` (the documented single source of
   truth) and have the test import them. **Owner-managed** — touches publish-guard config and risks
   flagging innocent tracked content; needs a deliberate owner false-positive pass.

## 10. Top risks

1. **Gate #3 is infra/execution-heavy, not code-heavy** — the net-new failover-load harness and a
   possible performance-tier miss are independent slip vectors. Treat benchmark runs as **bug-bearing**
   (the harness already found a real SQLite lock-free-read race), not a measurement formality.
2. **SQL Server active-passive has no per-row leases**, and `reclaims_inflight()==True` makes the engine
   *skip* the startup reset — no-split-brain rests entirely on the on-promotion sweep + the
   self-fence-before-lease-expiry invariant. Unit-covered, **no live kill-mid-load proof** until the
   Gate #3 failover run.
3. **`in_pipeline` blind spot** — without the gauge, the no-loss check can read a stalled router/transform
   as "drained." Hard predecessor of any published zero-loss baseline.
4. **Stale docs actively mislead new sessions into rebuilding finished work** — low effort, high leverage;
   do WS-DOCS first.
5. **Committed-benchmark PHI landmine** — any tracked result must be metrics-only and pass the publish
   denylist guard.
6. **Certificate lifecycle** is now an operator burden with no expiry alerting shipped — Q5(c) cert-expiry
   AlertSink closes this for v0.1.
