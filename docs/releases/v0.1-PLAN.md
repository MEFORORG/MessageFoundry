## Overview

v0.1.0 is MessageFoundry's **enterprise / HA milestone**. It commits to a **server database as the
supported production path — on *both* PostgreSQL and SQL Server** — with **engine high-availability
via an active-passive (primary + failover) model**, **native TLS**, and **no backend shipped under
an "experimental" label**.

The cut-line, given that promise:

- **Two production server-DB backends.** PostgreSQL is already production-grade for single-node
  ([store/postgres.py](../../messagefoundry/store/postgres.py): full staged pipeline,
  `supports_ingest_stage=True`, advisory-lock concurrency fixes, AES-256-GCM at-rest + rotation,
  retention). **SQL Server is *promoted* to production** — the single largest item in the release: it
  has **no staged pipeline today** (`supports_ingest_stage=False`; every staged method is a
  `NotImplementedError` stub — the engine hard-refuses it at startup,
  [engine.py:281-286](../../messagefoundry/pipeline/engine.py)), so promotion means building the full
  `ingress→routed→outbound` pipeline in T-SQL plus the H-6/H-7/H-8/M-6 concurrency fixes, retention,
  encryption parity, and a leadership primitive (`sp_getapplock`).
- **SQLite stays the supported single-node / dev default**, unchanged.
- **Engine HA is active-passive** — the Corepoint/Rhapsody model (see
  [ha-failover-research-2026-06-14.md](../marketing/ha-failover-research-2026-06-14.md)): only the
  **primary (leader)** processes; a **standby** takes over on failure. **DB HA is delegated to the
  DB tier** (PostgreSQL replication, SQL Server Always On) — not built by MessageFoundry.
- **Active-active horizontal scale-out is *parked* for 0.2.** MF's Track B code is built toward
  active-active (the Mirth-Platinum *scale-out* model), which carries the harder distributed-
  correctness risk and two verified bugs (below). For v0.1 we run that machinery in a **constrained
  active-passive mode** and make full active-active the 0.2 headline.
- **Native TLS**, not just a reverse-proxy posture: in-process API TLS + MLLP-over-TLS, so console/IDE
  and partners can reach a node directly.
- **Greenfield server-DB only** — no in-place SQLite→server-DB data migration in v0.1 (documented;
  operators drain SQLite before cut-over).
- **The four hard gates remain blockers.**

**Honest scope note:** promote-SQL-Server **+** native TLS **+** active-passive HA is roughly double
the workflow's original recommended cut-line. SQL Server staging (XL) and the throughput
characterization across three backends (XL) dominate the schedule; this is a multi-month release.

### Two verified correctness findings (confirmed against the working tree)

These drove the active-passive decision and remain on the books:

1. The per-row **`renew_leases` heartbeat is unwired** — defined at
   [store/postgres.py:1493](../../messagefoundry/store/postgres.py) with **zero call sites**. In
   *active-active* this is a live-row-theft bug. **Active-passive sidesteps it** (no live sibling
   processes concurrently) — a core reason for the pivot. The active-active fix defers to 0.2.
2. The **startup dead-letter sweep is lease-blind and runs on every node restart**
   ([engine.py:291-302](../../messagefoundry/pipeline/engine.py), comment confirms "*can kill a row
   another node is actively leasing*"). Under active-passive this must still be made leader-/lease-
   aware so a restarting standby cannot dead-letter the primary's in-flight rows.

Note: lease-gated startup recovery is **already committed** ([engine.py:264](../../messagefoundry/pipeline/engine.py)) — not a gap.

## Owner decisions

**Locked (binding):**

- **v0.1 goal:** enterprise / HA milestone.
- **Deploy target:** server DB as the supported production path, on **both PostgreSQL and SQL
  Server**; **no backend ships labeled "experimental."**
- **Engine HA:** **active-passive** (primary + failover). **DB HA delegated to the DB admins**
  (PostgreSQL replication / SQL Server Always On).
- **TLS:** **native in-engine** (API TLS + MLLP-over-TLS).
- **Authentication scope:** v0.1 authN is **local + Active Directory password bind (LDAP
  simple-bind)**. **Passwordless Windows SSO (Kerberos/SPNEGO) is in-tree but experimental, off by
  default (`kerberos_enabled=false`), and deferred to 0.2** — its production hardening (CI coverage,
  full SPNEGO handshake, keytab/SPN preflight) lands in 0.2 alongside federated SSO; no v0.1 surface
  claims it as production. Transport
  security is **native TLS** (API + MLLP-over-TLS, **server-identity only**). **Federated SSO
  (OAuth 2.0 / OIDC / SAML via Entra), TOTP MFA, and mTLS *peer/client* authentication defer to 0.2**;
  **SMART on FHIR is *Later*, gated on first adding a FHIR transport** (none exists today). Because
  Gate #4 makes the engine off-loopback-capable in v0.1, off-loopback admin therefore relies on
  **AD/Entra IdP-enforced MFA** (or an MFA-terminating reverse proxy) for remote access, and
  **local-password admins should stay on loopback** until first-party MFA lands in 0.2 (the
  single-factor-remote / ASVS 6.3.3 gap that
  [ADR 0002](../adr/0002-phase2-transport-security-and-strong-auth.md) names).
- **SQLite→server-DB migration:** **greenfield only** for v0.1.
- **Hard gates (all block `0.1.0`):** (1) PHI log redaction; (2) no experimental backends;
  (3) published throughput + tuning baseline; (4) native off-loopback TLS.

**Decisions taken this round — defaults; flag ⚑ to change:**

- **⚑ Throughput acceptance bar:** the tag requires published, reproducible numbers that **meet** the
  `fanout-baseline` SLOs (**≥ 200 msg/s sustained, ACK p99 ≤ 50 ms, e2e p99 ≤ 5 s**) on the reference
  spec, *or* a documented owner sign-off accepting revised SLOs in the same commit. (Silently
  lowering an SLO to the measured value is forbidden.)
- **⚑ Reference hardware:** baseline runs on an **8 vCPU / 32 GB / NVMe-SSD** host, **Python 3.11**,
  **PostgreSQL 16** / **SQL Server 2022**, engine and DB on separate hosts. Adjust to your prod spec.
- **⚑ CI:** run the `postgres-store` and `sqlserver-store` service-container jobs **on PRs that touch
  `store/` or `pipeline/cluster.py`** (small per-PR cost; billing budget is in place), not post-merge
  only — so HA/store correctness gates the PR. Raise the clustered **`pool_size` floor to ≥ 3**.
- **⚑ Release signing:** **Sigstore (keyless OIDC)** for the signed tag + artifacts in `release.yml`
  (no key custody). GPG remains an alternative.
- **⚑ Legal:** CLA activation / `COMMERCIAL-LICENSE.md` / NOTICE / SPDX headers are a **parallel
  track that does not block the code tag** (solo developer, no external contributors in flight).
- **⚑ Console scope:** the minimum HA-operability subset (primary/leader status + off-thread polling)
  is sufficient for v0.1; the Dead Letters GUI page, Alerts page, and multi-engine switcher defer to
  0.2 (CLI/API equivalents exist).

## Versioning & GA gate

- **Launch as `v0.1` — "Early Access."** The release is feature-complete, but the independent
  external **code review + penetration test** are performed **after** it ships. The `0.x` line plus
  the Early-Access label set honest expectations for a PHI engine shipping pre-audit — until v1.0, do
  **not** market it as GA / production-certified / independently-secured.
- **`v1.0` = the security-certified GA**, gated on: the external code review + penetration test
  complete and findings remediated, **plus** the off-loopback/TLS hardening (Gate #4). v1.0 is the
  "independently security-reviewed + penetration-tested" milestone.
- Aligns with the security plan's intent (an independent ASVS-L2 review + a DAST pass before the first
  production / off-loopback release).

## Hard release gates

Blockers. The tag cannot cut until every acceptance item is objectively met.

### Gate #1 — PHI log redaction

`redact()`/`safe_exc()` ([redaction.py](../../messagefoundry/redaction.py)) is used at the
Router/Handler exception sites; python-hl7's PHI-prone loggers are silenced and a
`ControlCharScrubFilter` runs on every record ([logging_setup.py](../../messagefoundry/logging_setup.py)).
Residual work:

- Install a `logging.Filter` that `redact()`-scrubs **formatted exception text including chained
  `__cause__`/`__context__`**, covering every outer-loop `log.exception()` site — delivery/router/
  transform catches and the `_on_*_worker_done` callbacks ([wiring_runner.py](../../messagefoundry/pipeline/wiring_runner.py)),
  the file/database/remotefile pollers, **and the cluster paths** (the leader reclaim sweep
  [leader_tasks.py](../../messagefoundry/pipeline/leader_tasks.py) + election/heartbeat loops in
  [cluster.py](../../messagefoundry/pipeline/cluster.py)). **Prefer the global filter** so future
  cluster `log.exception` calls are protected by construction.
- **SQL Server is now a production backend**, so its error/`last_error` paths must apply the same
  redaction discipline (not deferred as "preview").
- End-to-end pytest: ingest a synthetic ADT with a known PID name + MRN, force a Router/Handler
  exception **and** a delivery failure, assert **no record at WARNING+** contains those values.
- `serve` startup guard refusing/CRITICAL-warning on DEBUG logging in a `prod` environment.
- Write the acceptance spec into [docs/PHI.md](../PHI.md) §7.

**Acceptance:** the end-to-end test is green across parse→route→transform→deliver and the cluster
leader-sweep path; a chained exception carrying an HL7 body yields no body fragment in any traceback;
`serve` refuses DEBUG in `prod`; PHI.md §7 records the criteria.

### Gate #4 — Native off-loopback TLS

- **Close the MLLP/inbound bind-guard hole:** a non-loopback `[inbound].bind_host` is refused unless
  TLS is configured or an explicit insecure opt-in is passed ([__main__.py](../../messagefoundry/__main__.py)),
  mirroring the API `--allow-insecure-bind` guard.
- **Native API TLS (WP-13a):** `tls_cert_file`/`tls_key_file`/`tls_min_version` on `ApiSettings`,
  wired to `ssl_certfile`/`ssl_keyfile` in `uvicorn.run` (HSTS already activates over https,
  [api/app.py](../../messagefoundry/api/app.py)).
- **MLLP-over-TLS (WP-13b):** SSL context in `MLLPSource.start()` / `MLLPDestination.send()` with
  per-connection TLS on the `MLLP()` factory ([transports/mllp.py](../../messagefoundry/transports/mllp.py)).
- Also support `tls_terminated_upstream` + `trusted_proxies` for proxy-fronted deployments, but
  native is the supported default.
- Write the deployment doc enumerating **every off-loopback channel** in the active-passive topology
  (API edge, console→node, IDE→node, store-DB connection, MLLP inbound, the failover VIP) and each
  one's TLS posture.

**Acceptance:** uvicorn serves https with a configured cert and emits HSTS; MLLP accepts `ssl=` and
rejects plaintext when TLS is required (tests); a non-loopback plaintext bind is refused; the
deployment doc covers every channel.

## In scope

### Workstream A — Engine HA (active-passive primary/failover)

**Rationale.** The required HA model (Corepoint/Rhapsody-style failover), built as a constrained mode
of the existing cluster machinery. Reuses the committed leader election + leader-gated reclaim sweep;
avoids the active-active correctness landmines.

| # | Item | Effort |
|---|------|--------|
| A1 | **Leader-gate the whole processing graph.** Only the primary (leader) binds listeners + runs workers; the standby stays warm (contending for leadership only). Today listeners run on *every* node ([mllp.py:309](../../messagefoundry/transports/mllp.py)/`tcp.py`), so this is the central change: start/stop the graph cleanly on leadership acquire/lose. | L |
| A2 | **Leadership lease with self-fencing** (the split-brain guard — app-layer analogue of Rhapsody SBD). The old primary must **stop processing** if it cannot renew its leadership lease within the TTL, *before* a standby can acquire it (TTL > self-fence timeout). PostgreSQL: advisory lock / lease row; SQL Server: `sp_getapplock`. | L |
| A3 | **Make the startup dead-letter sweeps leader-/lease-aware** so a restarting standby never dead-letters the primary's in-flight rows ([engine.py:291-302](../../messagefoundry/pipeline/engine.py)). | M |
| A4 | **Failover recovery:** on promotion, the new primary recovers the dead primary's in-flight rows via the (committed) leader-gated reclaim sweep; confirm + test. | M |
| A5 | **`GET /cluster/status`** (read-only, permission-gated): current primary identity, node roster, per-node last-seen, lease state. No `/cluster` routes exist today. | M |
| A6 | **Failover test suite:** kill the primary mid-processing → standby promotes, binds listeners, recovers in-flight rows; **no double-processing under a simulated partition** (old primary self-fences); a routine standby restart does **not** dead-letter the primary's rows. | M |
| A7 | **Document the operational requirements:** a **floating VIP / LB health-check** for client reconnect (required, like Rhapsody/Corepoint); NTP/clock-skew floor vs lease TTL; the active-passive topology, in [docs/CONFIGURATION.md](../CONFIGURATION.md) + the deployment doc. | S |

**Acceptance:** with two nodes on a shared server-DB, exactly one processes; killing it promotes the
standby and recovers in-flight rows; a partitioned old-primary self-fences (no double delivery);
`GET /cluster/status` reports the primary + roster; the VIP + NTP requirements are documented.
**Depends on:** B (the backend providing the leadership primitive). **Effort: L.**

### Workstream B — Server-DB production-readiness (Postgres GA + **SQL Server promotion**)

- Author **docs/DEPLOY-SERVER-DB.md** (Postgres + SQL Server): connection settings, schema bootstrap,

**Acceptance:** a default `serve` against Postgres **or** SQL Server starts with no "experimental"

run fully in parallel — no dependency on A/B.

**Phase 2 — Throughput + console.** **D** runs after B (all backends final) and A (failover path), so
published numbers reflect the shipped config; it's the XL critical-path item. **G** runs in parallel
(consumes A5's `/cluster/status`).

**Phase 3 — Release.** **F** last (packaging describes the merged feature set); **owner runs the tag
+ mirror push.**

**Critical path:** Phase 0 → **B (SQL Server staging, XL)** → **A (active-passive)** → **D (3-backend +
failover benchmark)** → F. SQL Server promotion and the throughput characterization set the date.

## Risks

- **SQL Server promotion is XL and unstarted** — a correct T-SQL staged pipeline + the four
  concurrency fixes + retention + `sp_getapplock` leadership is the largest, riskiest single
  workstream and gates Gate #2, the SQL Server benchmark, and SQL Server active-passive. If it slips,
  the tag slips. Fallback to consider: ship SQL Server in a fast-follow `0.1.1` (would reopen the
  "both backends at tag" promise — Open Question 1).
- **Active-passive split-brain is the core correctness risk.** Without correct leadership-lease
  self-fencing (A2), a partitioned old primary + a promoted standby double-process. The fencing
  semantics (TTL > self-fence timeout, clock-skew bounds) and the partition test (A6) are the proof;
  a happy-path-only A6 can falsely pass the gate.
- **Failover is not instantaneous** — like Rhapsody (~minutes) there is a promotion window where MLLP
  senders see connection drops and reconnect via the VIP. Quantify it from the failover benchmark; it
  is not zero-downtime.
- **The load harness will surface more bugs under contention** — it already found a real store
  lock-free-read race (#200). Budget the 3-backend + failover runs to *fix*, not just measure.
- **Native TLS surface** — cert/key handling, min-version, and MLLP-over-TLS interop with partner
  senders add testing surface; mis-set ciphers/min-version can break a partner connection.
- **DB-failover reconnect (delegated DB HA)** — verify MF survives an actual Postgres-replica /
  Always-On failover (pool reconnect + transaction re-run), not just a dropped connection.
- **Gate-#2 `grep` self-failure** if the SQL Server / DATABASE / `cluster.py` relabels are incomplete.
- **Gate-#1 traceback leaks** via exception chaining, including the cluster leader-sweep path; the
  global redaction filter + the end-to-end test are the safeguards.
- **Greenfield-only data loss** if an operator points at a fresh server DB expecting their SQLite
  history; the release notes + deploy doc must state "drain SQLite first" plainly.
- **Release automation unbuilt** (no `release.yml`/tags; SBOM advisory + unpinned `@v4`); a rushed
  manual tag fails the project's own [RELEASE-GATE.md](../security/RELEASE-GATE.md).
- **Concurrent development** — another session is shipping into this repo (PRs #200/#201, worktrees);
  coordinate branches, worktrees, and memory writes to avoid contention.
- **Total scope** — three big builds (SQL Server, active-passive, native TLS) + an XL benchmark make
  this a multi-month release; breadth is the biggest schedule risk.
