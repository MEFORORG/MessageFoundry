# ADR 0090 — Resend a stored message to an alternate outbound connection

**Status:** **Accepted (2026-07-11, owner-ratified)** — BACKLOG #123 (resend-to-alternate) **and**, by the **§9 amendment (2026-07-11, owner-ratified)**, BACKLOG #153 (edit-and-resubmit). The §9 extension reuses this seam; it is no longer deferred.
**Deciders:** owner (ratifies) + reliability/store working group
**Related:** builds on [ADR 0001](0001-staged-pipeline-architecture.md) (the staged `ingress→routed→outbound` pipeline this injects an outbound row into) and its **finalizer sole-authority** rule; the **per-lane strict-FIFO** invariant (#285, [ADR 0059](0059-seq-only-per-lane-fifo.md) seq-only ordering, [ADR 0060](0060-rename-based-fifo-index-migration.md)); the **at-least-once / count-and-log** invariants (CLAUDE.md §2); the H2 idempotency ledger (`delivered_keys`, the shape `resend_log` copies); the store `replay()` **re-queue status exception** (the documented carve-out where a store method writes `messages.status` directly rather than the finalizer); [ADR 0073](0073-ownership-scoped-recovery-single-consumer-lanes.md) (single delivery consumer per outbound lane + owner-only outbound controls — the ownership gate this reuses); retention (`purge_message_bodies`, #34 / [ADR 0027](0027-per-connection-retention.md)); RBAC (`docs/security/PHASE-8C-RBAC.md`, per-channel scope) + step-up ([ADR 0077](0077-action-bound-step-up.md), the `MESSAGES_REPLAY` sibling this parallels).
**Code references** are `resend` branch tip at authoring; module paths are stable, line numbers approximate.

## 1. Context — an operator needs to re-transmit an already-sent message to a *different* partner

Every message MessageFoundry delivered is retained with its transformed outbound body on the `done` outbound row ([ADR 0001](0001-staged-pipeline-architecture.md); surfaced by `GET /messages/{id}/outbound`, #14). Operators already have `replay` (re-deliver a message to the destination(s) it was **originally** routed to). What they lack is the Mirth/Corepoint "resend to a different connection" affordance: a partner's endpoint changed, a downstream system needs a one-off copy, or a delivery must be diverted to a standby interface — **re-transmit the exact bytes we sent, but to an outbound connection the message was not originally routed to.**

This is BACKLOG #123. It is a small user-facing feature riding on top of the most invariant-dense seam in the system (the transactional staged queue that guarantees at-least-once + strict per-lane FIFO), so the hard part is not the endpoint — it is not breaking those invariants when a **second, control-plane writer** injects a row into a lane that until now had exactly one serial writer.

**Scope (owner-ratified):** #123 (§§1–8) = resend the **stored transformed body** to a chosen alternate outbound. **#153** (edit-then-resend) was originally deferred here but is now built as the **§9 amendment** on this same seam.

## 2. Decision

1. **Ship the stored transformed body — never re-run the transform.** A resend re-transmits *exactly what we sent*. Re-deriving the body by re-running the Handler could author new bytes (a changed handler, a changed `env()`), and would re-run the non-pure sanctioned lookups (`db_lookup`/`fhir_lookup`, ADR 0010/0043) whose results legitimately differ per pass. The source of truth is a **retained outbound-stage row's payload** for the origin message (dereferencing a shared body if the row carries a `body_ref`). **Any** retained outbound row is an eligible source regardless of its own delivery status — a `done`/`cancelled` body, a `dead` one (diverting a permanently-failed delivery to a standby interface is a marquee use case, §1), or a still-`pending`/`inflight` one all carry the same already-transformed bytes. `from_destination` in the outcome/audit therefore names the **source lane** the bytes were produced for, **not** an assertion that that lane itself delivered; `ResendSourceNotFound` fires only when no such outbound row exists at all (ERROR/FILTERED/UNROUTED, or the named `from_` has none).

2. **One new `stage='outbound'` queue row on the ORIGIN `message_id`,** targeting the alternate `destination_name`, landing at the alternate lane's **TAIL**, via a new store method `resend_to(...)`. It reuses `_insert_outbound_row` (+ `_reserve_shared_body` if ever fanned) and inlines the copied body. No new message id — the resend is another delivery of the same logged message (count-and-log intact).

3. **Idempotency** via a caller `idempotency_key` and a new **additive** `resend_log(resend_key UNIQUE)` table on all three backends, shaped like `delivered_keys` (hashes/ids only, no PHI, not ciphered). The outbound-row insert is **conditional on the `resend_log` insert actually creating a row** — a repeat under the same key is a no-op that reports the prior outcome; a **new** key is a genuine second resend.

4. **RBAC:** a **new** dedicated step-up permission `Permission.MESSAGES_RESEND` (distinct from `MESSAGES_REPLAY`; added to operator + administrator role sets). The endpoint requires access to **both** the origin's channel **and** the alternate outbound's channel (both audited), so a channel-scoped operator cannot push PHI to a partner they otherwise can't reach.

5. **Endpoint `POST /messages/{id}/resend`** beside `replay`; `engine.resend(...)` wrapper beside `engine.replay` that calls the store then `notify_work()` to wake the alternate lane's worker; audit a `message.resend` record (actor, from→to) — **never the body**.

## 3. The crux — FIFO under a second, control-plane writer (the writer-funnel)

**The invariant at risk.** Per-lane strict FIFO ([ADR 0059](0059-seq-only-per-lane-fifo.md), #285) rests on ordering a lane's live pending rows by `seq` (SQLite `rowid` / Postgres `BIGSERIAL` / SQL Server `BIGINT IDENTITY`), which the claim reads oldest-first. `store.py`'s `claim_next_fifo` docstring states the load-bearing premise: `ORDER BY seq` == receive order **only because there is exactly ONE serial writer per (stage, lane-key)**. `seq` is assigned at **INSERT**, *before* commit. A resend from an API node is a **second writer** to the alternate outbound lane. The inversion:

> Producer transaction `P` has inserted an older row into lane `L` (assigned `seq = k`) but has **not yet committed**. A resend `R` inserts (`seq = k+1`) and commits **fast**. A claimer runs. On **Postgres** (`FOR UPDATE SKIP LOCKED` under MVCC) `P`'s uncommitted row is **invisible**, so the claimer takes `R` (`seq = k+1`) and delivers it. `P` then commits and delivers `seq = k` **after** `seq = k+1` — a strict-FIFO inversion, the resend jumping an older in-flight row.

**Per-backend safety analysis (the mechanism, precisely):**

- **SQLite — safe unchanged.** SQLite has one process and one writer: the process-wide `self._lock` totally orders every insert (producer *and* resend), so commit-order == `rowid`-order by construction. `resend_to` takes `self._lock` like every other writer; there is no second concurrent writer to race. *(The engine's localhost API is in-process with the engine, so on SQLite there is not even a second process.)*

- **SQL Server — depends on the claim path (serial claim safe unchanged; pooled claim is fan-in-weakly-ordered, not blocked).** There are **two** SQL-Server claim paths and the safety argument differs:
  - **Serial per-lane claim (`claim_next_fifo`) — safe unchanged (by the #285 mechanism).** It reads the head `WITH (UPDLOCK, ROWLOCK)` and deliberately **without `READPAST`** (the #285 fix). A `TOP (1) … ORDER BY seq` scan that would include an older row a producer inserted-but-not-committed **blocks** on that row's `X` lock (it does *not* skip it, and does *not* read a phantom-free snapshot past it — `UPDLOCK` forces locking semantics even under RCSI). `IDENTITY` assigns `seq` in insert order under its latch, so the resend can never be assigned a lower `seq` than an *earlier* insert, and the claimer head-of-line-**blocks** on any lower-`seq` uncommitted row until it commits — then claims the true head. Under this path a resend cannot be delivered ahead of an older in-flight row; no extra lock is required.
  - **Pooled claim (`claim_fifo_heads`, [ADR 0066](0066-pooled-stage-claimers.md)) — the scale-out delivery path — is *not* blocked; it is weakly-ordered fan-in.** Its head-discovery is a **plain RCSI snapshot read** that, on multi-writer fan-in `destination_name` lanes, deliberately **adopts Postgres visibility**: a producer's *uncommitted* seq-N row is invisible and a committed seq-N+1 is claimable (the claim's own docstring documents this, sanctioned by ADR 0066's "no honored cross-inbound receive order" doctrine). So a fast-committing resend **can** be claimed ahead of an unrelated, still-uncommitted producer row in a shared fan-in lane — the exact MVCC/skip inversion the Postgres funnel prevents, and it is **not** prevented on SQL Server pooled mode. This is **within** ADR 0066's already-accepted weak cross-source fan-in ordering (the resend is a deliberate out-of-band act landing at the lane TAIL, and per-source FIFO — two rows from the SAME source — is never inverted, by the serial-writer argument), **not** a new invariant break. The earlier revision of this ADR overclaimed an absolute "a resend cannot be delivered ahead of an older in-flight row" for SQL Server; that holds for the serial claim only.
  - **Asymmetry (documented, owner-ratified).** Postgres is *strengthened* to a defined per-lane commit order by the advisory write-funnel (below); SQL Server pooled mode instead **relies on ADR 0066's fan-in doctrine**. A stricter symmetric fix — a per-lane `sp_getapplock` on every outbound producer + `resend_to` — is **deliberately declined**: it would add contention on the identified pooled-claim throughput wall (see the throughput work; the walls are claim-side) for a guarantee ADR 0066 does not make for cross-source fan-in. `resend_to` therefore takes only its per-key idempotency `sp_getapplock`, not a per-lane lock. If a future feed requires strict cross-source order into a shared alternate lane on SQL Server pooled mode, that funnel is the sanctioned addition.

- **Postgres — needs the funnel, and gets it via a per-lane advisory lock.** `FOR UPDATE SKIP LOCKED` + MVCC invisibility is exactly the vulnerable combination above. **Fix:** every writer of a lane's outbound rows takes a **transaction-scoped `pg_advisory_xact_lock` in a new lock class (`_LOCK_CLASS_OUTBOUND_LANE`) keyed by `destination_name`**, held from before `seq` assignment through commit. `resend_to` takes it for its `to` lane; the outbound-producing paths (`handoff`, `transform_handoff`, and the legacy single-step `enqueue_message`) take it for each distinct destination they insert into, **in sorted order** (deadlock-free). Under the lane lock, insert-and-commit for a lane is serialized, so **commit-order == `seq`-order** and no lower-`seq` row can be uncommitted-and-invisible while a higher-`seq` row is claimable. The lock is acquired **before** the per-message finalize advisory lock any of those methods later take (lane-lock < finalize-lock is a total order across all methods — methods that take only the finalize lock never take a lane lock, so no cycle exists).

  *Cost/scope note:* `pg_advisory_xact_lock` is an in-memory hash entry (~µs); it serializes concurrent **fan-in** producers to one destination lane (previously an accepted-as-benign undefined commit order) into a defined commit order — a strictly *stronger* FIFO guarantee, not a weaker one. The handoff path is not the identified throughput wall (the walls are claim-side — pooled-claim tempdb, ADR 0066). The Postgres/SQL Server backends are not runnable in this worktree; the **CI win2025 SQL-Server leg and the Postgres CI leg are the authoritative 3-backend gate** for this change.

**Original-order honesty (must not overclaim).** There is **no** `sequence_key`/ordering column in the store; an outbound lane is keyed by `destination_name` alone. A resend therefore lands at the destination-lane **TAIL** and *can* be delivered **after** newer same-partner messages already queued to that lane. There is **no "sequence-key sub-lane"** and this ADR does not claim one. The funnel guarantees only that the resend does not jump **ahead** of an older row already in the lane (no inversion of existing rows); it does **not** re-insert the resend into some historical position. **Decision:** we **do not block** a resend into an active ordered stream (blocking would require the ordering metadata we don't have, and the operator resend is a deliberate out-of-band act). The API response and the audit record document the tail-placement semantics; the console affordance (residual) will warn.

## 4. Idempotency + concurrent double-post gating (must run first, insert conditional)

`resend_log` mirrors `delivered_keys`: `resend_key TEXT PRIMARY KEY` (the caller idempotency key), plus `message_id`, `to_destination`, `from_destination`, `outbox_id` (the row it created, for reporting a duplicate's prior outcome), `created_at`. No body, no PHI, not ciphered.

The gating order inside the **single** `resend_to` transaction, identical in shape on all three backends:

1. `INSERT INTO resend_log (…) … ON CONFLICT(resend_key) DO NOTHING` (SQLite `INSERT OR IGNORE`; SQL Server `INSERT … WHERE NOT EXISTS`) **first**.
2. If it created **no** row (rowcount 0 / `RETURNING` empty), the key was already used: **read back** the prior row and return a `duplicate` outcome — **no** outbound row is inserted, `messages.status` is untouched.
3. Only if step 1 created a row do we read the source body, insert the outbound row, flip `messages.status`, and stamp `resend_log.outbox_id`.

Two API nodes racing the same key: exactly one wins the unique-key insert (same txn, on all three backends), and the loser's outbound insert never happens — **no double-send**. A repeat under a **new** key is, by design, a real second resend.

## 5. Retention interaction (empty-body reject + re-open race)

- **Empty-body-after-retention (blocker, must-fix #2).** `purge_message_bodies` nulls delivered outbound payloads (and `messages.raw`) in place on a fully-resolved message. `resend_to` reads the **source outbound row's** transformed body (deref'd); if that body reads back **empty** (`''` — retention-nulled, or the `mfb64`/inline body was blanked), `resend_to` **rejects with a 409** rather than shipping a zero-length body recorded as `PROCESSED`. A message with **no** outbound rows at all (ERROR/FILTERED/UNROUTED) has no source → a distinct 409 ("no delivered body to resend"). A retention-stripped-source test guards this.

- **Re-open race (major, must-fix #3).** Re-opening a purge-eligible `PROCESSED` message to `ROUTED` must be atomic w.r.t. retention's `NOT EXISTS(pending/inflight)` eligibility check. The status flip **and** the new pending outbound row commit in **one** transaction. Whichever commits first is safe: if `resend_to` commits first, retention's `NOT EXISTS(pending/inflight)` now sees the fresh **pending** row and skips the message; if `purge` commits first, `resend_to`'s source read sees the nulled `''` body and rejects (409, per above). The 409-on-empty-body guard is the load-bearing half; the same-txn atomicity closes the remainder. (SQLite serializes both under `self._lock`; the networked backends serialize via the per-row/lane locks each txn already holds.)

## 6. Finalizer sole-authority framing (must-fix #4)

`resend_to` writes `messages.status = ROUTED` (error cleared) **directly**, atomically alongside the new pending outbound row. This is framed as **the same documented re-queue exception `replay()` already uses** — not a claim that the finalizer is bypassed. The finalizer remains the **sole** authority for *terminal* dispositions: once the resend's outbound row resolves, `_maybe_finalize_message` recomputes `PROCESSED`/`ERROR` from all of the message's rows exactly as for any delivery. `ROUTED` is the correct intermediate ("a delivery is back in flight"), identical to `replay`'s outbound-only re-queue.

## 7. Target validation (must-fix #7)

Before accepting, the endpoint validates via the live `RegistryRunner` that the alternate outbound is a **registered** (`name in registry.outbound`), **owned-by-this-shard** (`destination_owner(name)` is `None` or this shard — ADR 0073; else 409 naming the owner), and **running** (`outbound_running(name)` — engine up, not operator-paused) connection. Otherwise the row would sit permanently `pending` and read as a silent drop. A graph-less engine (no runner) → 409. The `from_` source, when omitted, must resolve to exactly one delivered source row (else 409 "ambiguous — specify from").

## 8. Consequences

- **New capability:** operators can divert an exact re-transmission to an alternate partner, idempotently, RBAC- and step-up-gated, fully audited (no body).
- **Postgres hot path** gains one per-lane advisory lock on the outbound-producing handoffs. This *strengthens* fan-in FIFO and is off the known throughput walls, but is a real (if small) change to the reliability core — validated by CI's PG + SQL-Server legs, not locally.
- **Documented residuals:** (a) the **console/webconsole Resend UI** for #123 is a desired-if-clean follow-on; the API/engine capability is the deliverable of #123. (b) **#153 edit-and-resend** is now built — see the **§9 amendment** (web-console editor shipped; PySide6 desktop editor is the §9 residual). (c) tail-placement into an active ordered stream is **allowed** (documented), not blocked. (d) 3-backend parity for Postgres/SQL Server is asserted-and-CI-gated, not locally run.
- **Invariants preserved:** at-least-once (the resend row is an ordinary at-least-once outbound row; the source read is a pure copy of retained bytes); count-and-log (no new message; the origin stays logged with its evolving disposition); per-lane FIFO (§3 — strict on SQLite + the SQL-Server serial claim + Postgres-with-funnel; on the SQL-Server *pooled* claim the resend is weakly-ordered cross-source fan-in per ADR 0066, per-source FIFO preserved, tail placement); finalizer sole-authority for terminal state (§6); PHI (§4/§5 — ledger + audit carry hashes/ids only, never the body).

## 9. Amendment (owner-ratified 2026-07-11) — edit-and-resubmit a stored message (BACKLOG #153)

#123 resends the **stored** body unchanged. #153 (Corepoint operator-parity) lets an interface developer **edit certain messages before resending**: open a **copy** for editing, see it flagged **modified** on the first change, **resubmit** the edited body, **revert/cancel** the edit, and — critically — **the original message in the logs must NOT change**. This amendment extends this ADR's seam rather than opening a new one.

### 9.1 Ratified decisions

1. **The editable copy is CLIENT-SIDE + EPHEMERAL.** The edit lives only in the console/browser until Resubmit; **nothing edited is stored server-side as a draft** — no stored edited-PHI draft, no new retention/encryption surface. (The web-console editor GET renders the copy from the existing **audited raw view**; the desktop console edits an in-memory copy.) The edited body reaches the store only at the resubmit POST.

2. **Resubmit RE-ROUTES by default.** The edited raw **re-enters on the ORIGINAL message's channel as a fresh `RECEIVED` message** and flows the normal router→transform→outbound pipeline — processed like any receipt (re-parsed, re-validated, dead-lettered on failure, so a malformed edit is caught by the pipeline, not a synchronous NAK). A new store method `reingress(origin_message_id, raw, idempotency_key)` injects it as a pass-through-style child (ADR 0013 precedent). **Direct-to-a-chosen-destination is an OPTIONAL power path** that reuses #123's `resend_to` with the edited body (`body_override`), delivering straight to the alternate outbound.

3. **The ORIGINAL stays byte-identical (count-and-log).** Both paths create a **NEW, correlated child** (`correlation_id`/`correlation_root_id`/`correlation_depth` + an explicit `edited_from`); **the original message row is only READ, never opened for write.** The child links back to the original in the logs.

   *Clarification (2026-07-11, review #153-1/#153-2).* This is load-bearing for the **direct** path too, and is **not** a "resend on the origin `message_id`" like §2/§6. The direct `resend_to(body_override=…)` mints a **correlated child message** (status `ROUTED`, carrying the edited body + `correlation_id`/`edited_from` = origin) and hangs the alternate-lane outbound row off **the child's** id — so the **finalizer** recomputes the **child's** disposition (`PROCESSED`/`ERROR`) and **never** the origin's. Attaching the row to the origin `message_id` and flipping `messages.status`/`error` there (as §2/§6 do for **#123**, where the *same* body is legitimately re-delivered *as* the origin) is **wrong for #153**: the finalizer would then flip the origin (e.g. `ERROR`→`PROCESSED`) and the §6 status-write would erase its recorded error, mutating the original's audit/log — exactly what "the original must NOT change" forbids. The `from_destination`/audit stays ids-only; the origin gains only an `edit_resend` **event** row (an append-only log entry, not a write to the message row).

4. **Idempotent re-ingress — the enqueue_ingress double-deliver fix.** The naive fresh-receipt path (`enqueue_ingress`) mints a `uuid4` with **no dedup**, so a resubmit *retry* would DOUBLE-DELIVER. `reingress` instead mirrors §4's `resend_log` gating: the `idempotency_key` is claimed in `resend_log` FIRST (bound to `(origin, "@reingress:<channel>")` — a sentinel target disjoint from a resend-to-alternate's `(message_id, <outbound>)`), and the child message + ingress row are inserted **only when the claim created a row**; a retry under the same key is a no-op reporting the prior child (`duplicate`). Defense-in-depth: the child id is a **content-address of the key + channel + body** (explicit-id `INSERT-OR-IGNORE`), so even a partial-rollback re-run cannot double-inject. A key reused for a *different* origin is a `ResendKeyConflict` (409), never a silent drop.

5. **RBAC + step-up.** A new dedicated `Permission.MESSAGES_EDIT` (step-up, grant-audited; added to Operator + Administrator). It **implies `messages:view_raw`** (the edited body IS PHI and the client fetches the copy over the raw view) — realized by co-granting `view_raw` in every built-in role that holds `edit`. The DIRECT `to` power-path additionally requires per-channel access to the alternate outbound's channel (same cross-channel gate as #123).

6. **PHI-safe surfaces.** The edited body is PHI: it is **never logged at INFO+**, **never echoed in a 4xx** (a global `RequestValidationError` handler strips the offending `input`/`ctx` from the 422 detail so a malformed edited body can't surface in the validation response or an access log), and the `message.edit_resend` audit records **actor + original→new correlation only, never the body**.

### 9.2 API + UX

- **`POST /messages/{id}/edit-resend`** `{raw, idempotency_key, reroute=true, to?}`, `require_step_up(MESSAGES_EDIT)`. `reroute` (default, no `to`) → the idempotent re-ingress on the origin channel (`status="resubmitted"`, returns the correlated `new_message_id`); `to` set → the direct edited delivery via `resend_to(body_override=...)` (`status="resent"`). Both `duplicate` on a key retry. `engine.edit_resend_reroute` / `edit_resend_direct` wrap the store + `notify_work`.
- **Web console (required):** a message-detail **"Edit & resubmit →"** LINK opens a dedicated editor page (a step-up **unlock** GET, so the body-carrying POST submits inside a fresh window — mirrors the admin create-user form) with a `<textarea>` **COPY** of the raw, a **"Modified"** badge app.js reveals on the first change, a **Revert** button that restores the pristine copy, and a **Resubmit** button. The detail page's original raw is untouched. Newline note: a browser textarea normalizes newlines, so the editor compares CR/CRLF-insensitively; the resubmitted body is re-parsed tolerantly by the pipeline.
- **Desktop console (desired-if-clean):** the PySide6 message view is the offscreen-Qt-tested surface; the same open-copy/modified/revert/resubmit affordance is a mirror. **Residual if not built in this pass** (the web console is the ratified deliverable).

### 9.3 Consequences / residuals

- **Preserves every §8 invariant.** The re-route child is an ordinary fresh receipt (count-and-log intact — the origin is unchanged and stays logged); the direct path is an ordinary outbound row on its **correlated child** (§9.1.3) — the same §3 FIFO writer-funnel analysis applies (the lane is keyed by `destination_name`, independent of which `message_id` owns the row), while the origin stays read-only. No new PHI-at-rest surface (client-side ephemeral draft). Idempotency reuses the §4 ledger.
- **3-backend parity** (`reingress` + `resend_to(body_override=...)` on SQLite/Postgres/SQL Server) is asserted and **CI-gated** (the win2025 SQL-Server leg + Postgres leg); only SQLite is locally run here. The **web-console + desktop-console + TypeScript IDE** legs are CI-gated (offscreen-Qt / ide) as well.
- **Documented residual:** the **PySide6 desktop-console editor** is desired-if-clean (the web console is the required surface); browser textarea newline normalization is inherent (re-parsed tolerantly).
