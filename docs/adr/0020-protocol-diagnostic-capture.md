# ADR 0020 — Protocol-level diagnostic capture (Corepoint "Protocol Data" + "Protocol Text")

- **Status:** **Proposed.** Drafted, awaiting sign-off — no code yet. Sequence the build **after** the tee-relay→MFA store-auth-schema stack lands (its `_migrate`/`_CIPHER_COLUMNS`/`reencrypt_to_active` edits are textually adjacent to ours in `store/store.py`); rebase the capture work on top so its `_SCHEMA` / `_migrate` / cipher-pass additions append cleanly after the auth lines.
- **Built:** nothing yet. This document is the design only.
- **ADR number:** **0020.** `0019` is already claimed (`docs/adr/0019-pluggable-keyprovider-hsm-kms-vault.md` exists on disk, staged, and is listed in `docs/adr/README.md` row 0019). The prompt's assumption that 0019/0020 were both free is wrong; the genuinely free numbers are 0020 and 0021.
- **Decision in one line:** a toggleable, bounded, PHI-gated **per-connection** protocol trace — raw transport frames ("Protocol Data") + a textual transport/handshake trace ("Protocol Text") — that writes **nothing durable in steady state** (a per-connection RAM ring only), flushing to a new sibling **`protocol_trace`** artifact table only on a **transport error event** or an **operator snapshot**, with a live WebSocket for "tcpdump-on-demand". OFF by default, time-boxed, encrypted at rest, RBAC-gated + audited on read.
- **Related:** [ADR 0001](0001-staged-pipeline-architecture.md) (the staged queue + pure-re-run invariant this must not break), [ADR 0013](0013-query-response-orchestration.md) (the `response` artifact-table pattern this clones: a sibling of `state`/`reference`, **not** a `queue` stage), [ADR 0019](0019-pluggable-keyprovider-hsm-kms-vault.md) (the store cipher seam these PHI columns ride), [CLAUDE.md](../../CLAUDE.md) §2/§9 (reliability + count-and-log + PHI invariants — *do not break*).

## Context

Corepoint surfaces two diagnostic views per connection that MessageFoundry has no analogue for: **Protocol Data** (the literal on-the-wire bytes) and **Protocol Text** (a transport/handshake/error trace). Today the engine captures **deframed** payloads (in `messages.raw`) and ADR 0013 captures a partner's **application reply**, but the literal wire bytes and the **pre-message** transport failures — a partial/oversize MLLP frame, a TLS accept failure, a peer reset before any message completes, an allowlist refusal — are logged at WARNING and then lost. Those are exactly the failures an operator reconciling a flaky partner connection needs, and they have **no `message_id`** (no message ever formed), so they cannot live on `messages`.

The inbound boundary contract is `InboundHandler = Callable[[bytes], Awaitable[str|None]]` ([transports/base.py:43](../../messagefoundry/transports/base.py)): a source hands the runner the **deframed** message and writes back the returned reply string. A runner-level tap at `_make_handler` ([pipeline/wiring_runner.py:666-675](../../messagefoundry/pipeline/wiring_runner.py)) therefore sees the deframed payload + the reply, but **not** the framing bytes and **not** a pre-message failure. The literal wire bytes and pre-message events are only visible **inside the connector's read loop** (e.g. `MLLPSource._on_client`, [mllp.py:570-635](../../messagefoundry/transports/mllp.py)). So a faithful Protocol-Data/Text feature needs **two tiers**: a runner-side proxy for the deframed-payload + ACK view (connector-type-agnostic), and a narrow connector byte-hook for the literal frame + pre-message events (socket connectors only).

The hard constraint is PHI. The HL7 lives **inside** the MLLP frame, and the AA ACK echoes MSH — so a captured frame and a captured ACK are both PHI and must be encrypted at rest, short-window-purged, RBAC-gated, audited on read, and never logged at INFO+ (CLAUDE.md §9). And the capture must be a **pure observer**: it must never block a listener/delivery lane, never raise into the MLLP `_on_client` catch-all (which `break`s the client connection, [mllp.py:615-622](../../messagefoundry/transports/mllp.py)), and never participate in any ingress→routed→outbound handoff transaction.

## Decision

### §1 Two tiers, one sink (RAM-first)

A new stdlib-only module `messagefoundry/transports/protocol_trace.py` (imports **nothing** from `pipeline/`/`store/`/`api/` — takes injected callbacks, the `leader_gate` precedent at [wiring_runner.py:348](../../messagefoundry/pipeline/wiring_runner.py)) holds a per-connection `ProtocolRing` (two bounded `deque`s — frames + events — plus a bounded live-WS subscriber set). Steady state writes **only** to RAM and ages out; near-zero PHI at rest.

- **Tier 1 — transparent runner-side proxy** (`TracingSourceProxy(SourceConnector)` / `TracingDestinationProxy(DestinationConnector)`): wraps the built connector with a `record` callback the runner injects. The source proxy records the deframed inbound payload (re-framed via the resolved codec, head-capped) + the ACK reply string as a `kind='data' direction='ack'` frame; the dest proxy records the outbound payload + the `DeliveryResponse` outcome. It MUST forward `polls_shared_resource`, the `leader_gate` kwarg of `start`, `test_connection`, `stop`, `capture_response` (read via `getattr` at [wiring_runner.py:388](../../messagefoundry/pipeline/wiring_runner.py)), `aclose`, and the `send` return value **verbatim** (so ADR 0013 capture + re-ingress are untouched), with a `__getattr__` fallback for future introspection.
- **Tier 2 — framing-owner byte hook** (socket connectors only — the motivating capture): each socket connector gains an OPTIONAL `on_protocol_event: Callable[..., None] | None = None` (default None = byte-identical), **set by the runner after build** (NOT a settings-dict value, NOT a builder arg — the `leader_gate` precedent). Call sites in `MLLPSource._on_client`: raw inbound `chunk` ([mllp.py:594/598](../../messagefoundry/transports/mllp.py)) → a `kind='data' direction='in'` frame of the LITERAL bytes (head-capped); the allowlist refuse ([mllp.py:582](../../messagefoundry/transports/mllp.py)), the `MLLPFrameError` over-cap ([mllp.py:607](../../messagefoundry/transports/mllp.py)), the catch-all ([mllp.py:615](../../messagefoundry/transports/mllp.py)), and a NEW `except ssl.SSLError` around the TLS accept → a `kind='text' direction='event'` line (peer + reason; NEVER a body); the framed reply ([mllp.py:605](../../messagefoundry/transports/mllp.py)) → a `kind='data' direction='ack'` literal-framed frame. Mirror in `MLLPDestination.send`, `transports/tcp.py`, and `transports/x12.py` source+dest. File/DATABASE/REST/SOAP have no socket frame → Tier-1 deframed capture only (honest: their "Protocol Data" IS the message body, already in `messages.raw`).

**Structural no-raise contract.** The module exposes `safe_record(ring, frame_or_event, *, on_subscriber=...)` that wraps the deque append + subscriber fan-out in `try/except Exception` (logs `safe_exc`, name+type only — never `head_hex`/`text`). Both `record` and `on_protocol_event` are thin shims calling `safe_record`, so a capture bug is **structurally incapable** of raising into the `_on_client` loop or a delivery lane. A fault-injection test injects a record callback that raises on every call and asserts the client stays connected, the message is still ingressed + ACKed, and the delivery lane still drains. (Resolves the data-path-effect hazard at mllp.py:615.)

### §2 Storage — a new `protocol_trace` table (sibling artifact, id-keyed)

Connection-keyed, NULL `message_id` is the point (pre-message failures have no message). It is **NOT** a `Stage` → invisible to `_maybe_finalize_message`'s `FROM queue ... GROUP BY stage, status` scan ([store.py:3583-3584](../../messagefoundry/store/store.py), verified) → it can never pin a message out of `PROCESSED` or flip it to `ERROR`. **id-keyed surrogate PK** so the two PHI columns ride the generic id-keyed cipher loops on SQLite/Postgres.

Columns (keyword-clean): `id` (surrogate PK), `connection` (filter axis), `direction` ('in'|'out'|'ack'|'event'), `kind` ('data' = frame bytes | 'text' = transport line), `message_id` (nullable FK, best-effort correlation), `peer` (redacted ip:port, NOT PHI), `nbytes` (pre-truncation count), `truncated`, **`frame_hex`** (head-capped hex frame bytes; PHI; encrypted; NULL once purged), **`trace_text`** (transport line; encrypted; NULL once purged), `captured_at`; index `ix_ptrace_conn(connection, captured_at)`. Names are `frame_hex`/`trace_text` (not `data`/`text`) because T-SQL `data`/`text` are keyword-fragile. A `kind='data'` row carries `frame_hex` with NULL `trace_text`; a `kind='event'` row the reverse.

Three backends, identical shape: SQLite `id INTEGER PRIMARY KEY AUTOINCREMENT` in `_SCHEMA` after the `response` block; Postgres `id BIGSERIAL PRIMARY KEY` under the schema advisory lock; SQL Server `id BIGINT IDENTITY(1,1) PRIMARY KEY` under `IF OBJECT_ID('protocol_trace','U') IS NULL` + the index under `IF INDEXPROPERTY(...) IS NULL`.

Store-protocol additions (`store/base.py`, all 3 backends): `supports_protocol_trace: bool = True` (mirror `supports_response_capture` + the fail-closed start gate at [wiring_runner.py:386-395](../../messagefoundry/pipeline/wiring_runner.py)); `append_protocol_trace(rows)` (batched multi-row INSERT in ONE transaction, OFF the message-handoff txn; a failure is swallowed + logged WARNING with connection name + `safe_exc` type only, NEVER rolls back ingress/delivery); `read_protocol_trace(...)` (decrypt at the boundary, DTO `CapturedTrace` mirroring `CapturedResponse`); `purge_protocol_trace(*, older_than)` (body-null `UPDATE ... SET frame_hex=NULL, trace_text=NULL WHERE (frame_hex IS NOT NULL OR trace_text IS NOT NULL) AND captured_at < ?`, returns rows actually nulled — the WHERE guard makes a second purge return 0, idempotent). `trace_text` is additionally passed through `safe_text()` (#120 chokepoint) before encryption; `frame_hex` is **not** (we keep the literal frame for diagnosis; it is RBAC + audit + short-window-purge + view-raw gated instead).

### §3 Capture path (engine-side; no api/console import; base.py NOT touched)

**`transports/base.py` is NOT modified** — wrapping happens only at the **live runner build sites** via small `_maybe_wrap_source`/`_maybe_wrap_dest` helpers in `wiring_runner.py`, right after `build_source(...)`/`build_destination(...)` returns: `start_inbound` ([~339](../../messagefoundry/pipeline/wiring_runner.py)), the dest build ([~385](../../messagefoundry/pipeline/wiring_runner.py)), and reconcile ([~579/584](../../messagefoundry/pipeline/wiring_runner.py)). The helpers, gated on the per-connection `protocol_trace` toggle AND the master switch, wrap the connector in the proxy with `record` bound to the connection's ring and set `connector.on_protocol_event` for socket connectors. **Explicitly EXCLUDED:** `build_test_connection` (a throwaway connector for the `test_connection` API) and the standalone `build_check`/dryrun path (pipeline-free by design) — a throwaway/validation connector has no ring and no runner and must never be traced. Because we never touch the `build_source`/`build_destination` registry seam, the dependency direction holds (`transports/` never imports `pipeline/`) and there is **zero** connector-type special-casing in `pipeline/`.

**Persistence trigger (RAM-first; flush only on error/snapshot).** `record(...)` appends to the ring (O(1), bounded) and fans out to live-WS subscribers — **NO DB call inline**. A single supervised, cancellable `_protocol_trace_flusher` asyncio task (sibling of the delivery workers, owned by RegistryRunner) drains an `asyncio.Queue` of flush requests and batches `append_protocol_trace` off the hot path. Flush is requested ONLY on (1) a transport **error** ProtocolEvent (over-cap / TLS-fail / reset / DeliveryError) → flush that connection's current ring window, or (2) an operator `POST /connections/{name}/protocol-trace/snapshot`. Queue overflow → drop-oldest + a `dropped` counter (bounds memory; never blocks the listener/delivery lane). **Crash-window caveat (accepted):** un-flushed RAM frames are lost on a hard crash — acceptable for a diagnostic trail (it is not a reliability surface); error-triggered flushes commit the failure context.

### §4 Config (per-connection OFF by default, bounded)

Following the `capture_response`/`capture_max_rows` precedent ([wiring.py:852-855, 880-882](../../messagefoundry/config/wiring.py)): add to the socket-bearing source/dest factories `protocol_trace: bool = False`, `protocol_trace_frames: int = 64`, `protocol_trace_lines: int = 256`, `protocol_trace_head_bytes: int = 128`. All go **verbatim into the settings dict** (plain JSON/TOML scalars — no callable ever lands in `settings`), so they are auto-authorable from `connections.toml` with zero `_INBOUND_KEYS`/`_OUTBOUND_KEYS` change. Cross-field guards at the shared chokepoint `build_inbound_connection`/`build_outbound_connection` ([wiring.py:1353/1503](../../messagefoundry/config/wiring.py), fires in dryrun + `messagefoundry check`): reject `protocol_trace=True` on a no-wire transport (Loopback/Timer) with a `WiringError`; clamp `frames ≤ 4096`, `lines ≤ 4096`, `head_bytes ≤ 4096`; and **PHI fail-closed** — if ANY connection sets `protocol_trace=True`, require `retention.protocol_trace_retention_hours > 0 and ≤ 168`, raising `WiringError` otherwise (an operator cannot leave raw-HL7 frames at rest indefinitely). Service-level: a new `[diagnostics].protocol_trace_enabled: bool = False` master kill-switch in `config/settings.py`, threaded `settings → __main__ → Engine → RegistryRunner` (mirror `max_correlation_depth` at [__main__.py:429](../../messagefoundry/__main__.py); a settings field NOT threaded is silently inert). The retention knobs `protocol_trace_retention_hours: int = 24` (0=off) and `protocol_trace_max_persist_rows_per_conn: int = 5000` (0=off, advisory) live on **`RetentionSettings`** (NOT `[diagnostics]`) so they thread through the existing `RetentionRunner(store, settings.retention, ...)` construction.

### §5 PHI encrypt + purge (per-backend, explicit)

**SQLite/Postgres:** add `('protocol_trace','frame_hex')` and `('protocol_trace','trace_text')` to `_CIPHER_COLUMNS` ([store.py:748-760](../../messagefoundry/store/store.py)). Their id-keyed loops in `_encrypt_existing_rows` and `reencrypt_to_active` already guard `NOT LIKE prefix AND {col} <> ''` (verified) → NULL/blank-safe, zero bespoke pass. Write `self._enc`, read `self._dec`.

**SQL Server** (verified: **HAS NO `_CIPHER_COLUMNS`, NO `_enc`/`_dec`**): its id-keyed `_encrypt_existing_rows` loop ([sqlserver.py:353](../../messagefoundry/store/sqlserver.py)) uses ONLY `WHERE {col} NOT LIKE ?` — safe for its NOT-NULL members but **unsafe for our nullable, blank-after-purge columns** (a literal `''` would be re-encrypted). So `protocol_trace` gets a **SEPARATE id-keyed pass** in BOTH `_encrypt_existing_rows` and `reencrypt_to_active` carrying `AND {col} <> '' AND {col} IS NOT NULL` (mirror the `response` separate pass at [sqlserver.py:381-411](../../messagefoundry/store/sqlserver.py), but `WHERE id=?`), plus inline `self._cipher.encrypt(v) if v else v` / decrypt on append/read (matching [sqlserver.py:1050-1051](../../messagefoundry/store/sqlserver.py)).

**Retention** ([pipeline/retention.py](../../messagefoundry/pipeline/retention.py)): `RetentionRunner.enabled` ([~97-109](../../messagefoundry/pipeline/retention.py)) gains `or s.protocol_trace_retention_hours` so a trace-only config still spawns the loop; `run_once` calls `purge_protocol_trace(older_than = now - hours*3600)` when configured; `RetentionPass` (frozen, [~43-65](../../messagefoundry/pipeline/retention.py)) gains `ptrace_purged: int` — added to the dataclass, the non-leader early-return positional construction, `did_work`, and every test constructor. Stays leader-gated (exactly one node purges).

### §6 API + RBAC + audit

Connection-scoped (NOT under `/messages` — these rows have no `message_id`): `GET /connections/{name}/protocol-trace?...` → `ProtocolTracePage`; `GET /connections/{name}/protocol-trace/live` (WebSocket, reuse the stats-WS infra, bounded subscriber + drop-oldest queue, RBAC re-checked on connect); `POST /connections/{name}/protocol-trace/snapshot`. New `Permission.PROTOCOL_TRACE_READ` in `auth/permissions.py` (code-only, NO migration — the roles table stores no permission columns; granted to **Operator**; Administrator auto-gets it via `frozenset(Permission)` — update the `test_auth_core.py:25` assertion). All routes gate `require_phi_read(Permission.PROTOCOL_TRACE_READ)`; channel scope → **404** out-of-scope via `_audit_channel_denied` ([app.py:261](../../messagefoundry/api/app.py)). **Both** PHI columns (`frame_hex`, `trace_text`) sit on the stricter **`MESSAGES_VIEW_RAW`** tier (the table is a raw-frame surface; `frame_hex` echoes the frame, and we keep `trace_text` on the same raw tier to avoid an under-gating mismatch if a transport line ever echoes message-derived content), registered in `api/field_authz.PHI_FIELDS` keyed on the EXACT type. Every read on all three routes emits `record_audit('protocol_trace.read', ...)` BEFORE returning; a non-NULL-`message_id` exposed body also `record_view` + the exposure coalescer. `docs/PHI.md` documents `protocol_trace.frame_hex` as a `MESSAGES_VIEW_RAW`-tier raw-PHI surface beside `messages.raw`, with the short-window-purge requirement.

## Consequences

**Positive**
- Corepoint's Protocol Data + Protocol Text become expressible per connection, including the pre-message transport failures that have no `message_id` and could never live on `messages`.
- **Pure observer.** Proxies forward `send`'s return + the ACK reply unchanged; capture writes sit OUTSIDE every ingress→routed→outbound handoff; a write failure is swallowed, never rolls back a message; routers/transforms untouched. `record`/`on_protocol_event` are structurally no-raise, so a capture bug can never drop an MLLP client.
- **Byte-identical when off.** `protocol_trace=False` (the default) → no proxy wrap, `on_protocol_event=None`, no ring, no DB row.
- Finalizer/disposition untouched — `protocol_trace` is a sibling artifact, not a `Stage`.
- `transports/base.py` registry seam untouched → zero connector-type special-casing in `pipeline/`; the dependency direction holds.

**Negative / costs**
- A new PHI-bearing table, read surface, RBAC permission, audit event, retention branch, live-WS, and a connector byte-hook in three socket transports — surface area across store/transports/pipeline/config/api/auth, each needing a test, hand-mirrored across three backends (SQL Server gets its own cipher pass).
- The raw-frame-at-rest surface is genuinely sensitive; mitigated by RAM-first (near-zero at rest in steady state), head-cap 128, fail-closed retention guard, view-raw gating, and read-audit — but it is a new raw-PHI tier and is documented as such in `docs/PHI.md`.
- A hard crash loses un-flushed RAM frames (accepted: diagnostic, not a reliability surface).

## Alternatives considered

| Alternative | Why considered | Why rejected | Verdict |
|---|---|---|---|
| **Sibling `protocol_trace` table, RAM-first, flush-on-error** *(chosen)* | Invisible to the finalizer; near-zero PHI at rest; captures pre-message failures with no `message_id` | New surface in six subsystems; a raw-frame-at-rest tier | **Adopted** |
| **Wrap at the `transports/base.py` registry seam** | One decorator covers every build path | Would trace throwaway test/check/dryrun connectors (no ring, no runner) and risks coupling `transports/` to `pipeline/`; the runner build sites are the cleaner, exclusion-able seam | **Rejected (→ wrap at the live build sites only)** |
| **Put `record`/`on_protocol_event` in `settings`** | Uniform with other per-connection config | A callable in `settings` breaks JSON/TOML-serializability and the `connections.toml` desugar; runtime injection is the `leader_gate` precedent | **Rejected (→ runtime injection after build)** |
| **Always-on durable capture (every frame to the DB)** | Simplest "it's always there" | Per-frame DB await on the hot path; unbounded PHI at rest; defeats the purge window | **Rejected (→ RAM-first, error/snapshot flush)** |
| **Reuse the ADR 0013 `response` table** | No new DDL | A protocol frame has no `(message_id, destination_name, response_seq)` identity and is usually message-less; overloading `response` would corrupt its per-destination authoritative-reply semantics | **Rejected (→ dedicated table)** |
| **`data`/`text` column names** | Mirrors Corepoint's labels | T-SQL `data`/`text` are keyword-fragile | **Rejected (→ `frame_hex`/`trace_text`)** |
