# ADR 0067 ‚Äî Persistent outbound MLLP connections (connection reuse for delivery)

**Status:** Accepted ¬∑ **Date:** 2026-07-02
**Relates-to:** ADR 0001 (staged pipeline ‚Äî the per-outbound delivery worker this rides on), ADR 0002/WP-13b (per-connection MLLP-over-TLS ‚Äî the outbound `SSLContext` reconnects must reuse), ADR 0013 (capture_response ‚Äî the ACK read on the reused connection), ADR 0021 (inbound ACK vocabulary), ADR 0066 (pooled per-stage claimers ‚Äî the companion WS-C fix on the *claim* side; this ADR is the fix on the *egress* side), PR #730 (`MLLPDestination._describe_error` ‚Äî errno/winerror detail every reconnect path must carry), docs commit 82aa790 (WS-C claim-storm finding + the interim TCP-tuning doc), [docs/SERVICE.md ¬ß "High-delivery-rate TCP tuning"](../SERVICE.md), [docs/CONNECTIONS.md ¬ß MLLP](../CONNECTIONS.md), CLAUDE.md ¬ß2 ("Reliability invariant" ‚Äî at-least-once, outbounds must be idempotent) + ¬ß8 (ACK/NAK conventions).
**Code references** are `origin/main @ 7531545`; line numbers are approximate ‚Äî locate exactly at implementation time.

## 1. Context

`MLLPDestination.send()` opens a **fresh TCP connection per delivered message**: connect (`mllp.py:446-459`) ‚Üí write frame + drain (`:461-462`) ‚Üí read one ACK (`:463`, `_read_ack` `:480-490`) ‚Üí close in a `finally` (`:468-473`). The class docstring says so explicitly ‚Äî "Phase 1 opens a fresh connection per delivery ‚Äî simple and robust to flaky peers; a persistent/pooled connection can come later" (`mllp.py:377-387`). This ADR is that later.

The forcing problem is measured, not speculative. The engine is the **active closer** of every per-message connection, so each delivery parks a socket in `TIME_WAIT` on the engine host. On the 2026-07-02 claim-storm bench (1,500 lanes, 525 msg/s aggregate, 2 engines), `TIME_WAIT` accumulation exhausted the default Windows ephemeral-port range (16,384 ports) and produced **~18.6k dead-letters** from `WinError 10048/10055`-class connect failures ‚Äî the partner was healthy the whole time. Those failures were initially *undiagnosable* because `str(exc)` was empty; PR #730 added `_describe_error` (`mllp.py:414-434`), which names the type and appends `errno`/`winerror`/`strerror` ‚Äî its own docstring cites the same ~18.6k-dead-letter incident. The interim mitigation is **host tuning** documented in [docs/SERVICE.md:303-329](../SERVICE.md) (`netsh ‚Ä¶ set dynamicport tcp start=22000 num=43535` + `TcpTimedWaitDelay=30`), which that section itself frames as a stopgap: "persistent connections are a planned change" (`SERVICE.md:305-306`).

A persistent connection per destination is the durable fix: it removes the per-message TCP handshake (and TLS handshake on `tls=true` outbounds) and the per-message `TIME_WAIT`, shrinking port pressure from *per-message* to *per-reconnect*. It also deletes 1 RTT + handshake from every delivery's serial per-lane chain (per-interface throughput is serial-lane-latency-bound). It is also the MLLP-standard posture ‚Äî MLLP peers (and Mirth/Corepoint outbounds) hold a connection open across messages by default.

**Invariants in play** (CLAUDE.md ¬ß2, verbatim): "the transactional staged queue on SQLite (WAL) gives at-least-once delivery, retries, replay, and dead-lettering"; "outbound connections must still be **idempotent**"; "Each outbound connection drains independently". And ¬ß8/CONNECTIONS.md:232-234: "an outbound delivery that is sent but whose ACK is lost ‚Ä¶ is retried, so the receiver may see a duplicate. This is the documented at-least-once trade-off ‚Äî outbound receivers must be idempotent." The same trade-off is already stated in the connector docstring (`mllp.py:384-387`).

**Load-bearing code facts** the design leans on:

- **The delivery worker is the lane's single consumer.** One worker per outbound (`_delivery_worker`, `wiring_runner.py:1793`), FIFO claims the due head only (`:1806-1814`), and ADR 0059/0061 assert "exactly ONE serial writer per (stage, lane-key)". So `send()` on one connector instance is already **serialized by construction** ‚Äî no pool, no send-side locking needed.
- **The lifecycle hook exists.** `DestinationConnector.aclose()` (`base.py:226-227`, default no-op) is already called on engine teardown (`wiring_runner.py:1130-1131`) and on reload reconcile for every replaced/removed connector (`:1287-1320` ‚Äî "the old one is closed (a single racing send ‚Ä¶)"). A persistent socket has a place to die.
- **Reload swap race is already documented.** "Connector + retry are re-resolved per item so a reload can swap an outbound's settings under us with at most one racing send (which fails + retries ‚Äî outbounds are idempotent)" (`wiring_runner.py:1826-1828`). Concurrent `aclose()`-vs-in-flight-`send()` is therefore an *existing, sanctioned* race with defined semantics (fail loud, charged, retried).
- **Failure taxonomy.** Transport `DeliveryError` ‚Üí `mark_failed` + backoff + edge-triggered `connection_lost` (`:1886-1893`); `NegativeAckError` = partner rejection on a *healthy* transport ‚Äî AR/CR dead-letters, AE retries, and it "never flips" lane health (`base.py:108-123`, `wiring_runner.py:316-319`, `:1875-1885`).
- **TLS outbound is supported** (`_mllp_ssl_context(s, server=False)`, `mllp.py:351-374`), context built **once** at `__init__` (`:412`) and passed to `open_connection` with SNI (`:450-453`). An `ssl.SSLContext` is reusable across connections, so reconnect needs no context rebuild.
- **`test_connection` never touches the live connector** ‚Äî the reachability probe is `probe_tcp_reachable` (`mllp.py:476-478`) and the API builds a *fresh* connector for tests (`wiring_runner.py:749-755`).
- **Windows Proactor close wedge (#55).** An unbounded `writer.wait_closed()` can hang forever on a pending overlapped op; the inbound side already bounds every close (`mllp.py:621-626`, `:722-729`). Persistent-connection close paths must reuse the bounded pattern.

## 2. Decision

**Give `MLLPDestination` one lazily-established, reused TCP connection per destination instance, with reconnect-before-first-byte as the only sanctioned internal retry, per-outbound `persistent` knob, an idle timeout, and close-on-`aclose()`.** Single connection, not a pool ‚Äî the delivery worker is the lane's single serial consumer, so a pool buys nothing and would break per-lane FIFO locality.

**Ship the knob `persistent=false` (connect-per-message, today's proven posture) as the default for this release; `persistent=true` (the reuse design below) is a documented opt-in.** The reuse design is fully built and tested ‚Äî only its *default* is held for one release. The default flips to `persistent=true` in a subsequent release once the concrete ¬ß8 trigger is met (a real deployment runs `persistent=true` on a live feed clean, plus a mid-transaction stray-frame test / the MSA-2‚ÜîMSH-10 correlation lands or the residual is field-confirmed benign). Rationale: the connect-per-message posture is the project's tracked charter default (BACKLOG #82.1); the forcing incident (~18.6k dead-letters) is synthetic-bench-only at 1,500 lanes / 525 msg/s, far above the Pilot/Standard tiers most on-prem adopters run; the conceded mid-transaction stray-frame correctness edge (¬ß7) is untested and the complete fix (BACKLOG #82) unbuilt; and reversibility is asymmetric ‚Äî `false‚Üítrue` later is a one-line default + release note, whereas `true‚Üípartner-reuse-intolerance-in-prod` is a live clinical feed already broken. The safety invariants (FIFO / at-least-once / count-and-log) hold either way, so defaulting to the proven path costs only per-message handshake/`TIME_WAIT` on the happy path ‚Äî acceptable at the tiers that never hit the bench regime, and those that do can opt in + apply the shipped host tuning (docs/SERVICE.md).

What it must **not** break: at-least-once + retry/backoff semantics (`DeliveryError` ‚Üí `mark_failed`, never an internal resend loop); `NegativeAckError` AR-fail-fast/AE-retry policy; `capture_response`/`reingress_to` (ADR 0013); `simulate` egress suppression (`send()` is simply never called ‚Äî `wiring_runner.py:1865-1872`); `encoding_characters` re-encode; per-lane FIFO; the inbound side (untouched); `_describe_error` detail on every failure (PR #730).

Concretely:

1. **Knobs on `MLLP()`** (`config/wiring.py:641-718` ‚Üí `settings` dict ‚Üí `Destination.settings`, `models.py:293-299`; rides `connections.toml` through the same factory desugar, ADR 0007):
   - `persistent: bool = False` ‚Äî outbound-only (inbound ignores it, like `timeout_seconds`). **Default `false` this release** (connect-per-message ‚Äî today's proven posture, BACKLOG #82.1); `persistent=true` is the documented **opt-in** reuse path (the MLLP-standard posture; Mirth/Corepoint keep-alive parity). The default flips to `true` per the ¬ß8 trigger ‚Äî a data-gathering step, not an open-ended hold.
   - `idle_timeout_seconds: float | None = 60.0` ‚Äî do not reuse a connection idle longer than this; `None`/`0` = never expire on idle.
   - `max_connection_age_seconds: float | None = None` ‚Äî optional recycle-by-age (LB/firewall hygiene); off by default.
   - `persistent=False` (**the shipped default this release**) reproduces today's connect-per-send **wire behavior** ‚Äî one connection per delivery, same dial/frame/ACK semantics, same error text ‚Äî the proven posture for the Pilot/Standard tiers and the compat mode for legacy LIS/interface engines that require connection-per-message. Two intentional hardening deltas from the legacy code, both deliberate: the per-send close is **bounded** (the #55 Proactor-wedge pattern; legacy awaited `wait_closed()` unbounded and could hang the delivery worker forever) and the fail-loud serial-`send()` assert (¬ß2.5) applies in **both** modes (concurrent `send()` on one instance was never a sanctioned pipeline use ‚Äî the worker is the lane's single serial sender).
2. **Lazy establish, reuse on success.** `send()` with no cached connection dials exactly as today (same `connect_timeout`, same `ssl`/SNI args ‚Äî a dial failure is a charged `DeliveryError` with `_describe_error`, zero bytes written). On a **fully successful transaction** ‚Äî frame written, one ACK read and parsed, *including a NAK* (a NAK is a complete request/response on a healthy transport) ‚Äî the `(reader, writer)` pair is cached with a last-used timestamp. **Any `DeliveryError` closes and discards the connection** ‚Äî a failed transaction never leaves a socket in an unknown framing state to bleed a late ACK into the next send. Desync guard, two halves: **in-transaction** ‚Äî if unread bytes remain after the ACK frame completes (same read run), close instead of reuse; **at reuse time** ‚Äî any bytes the peer sent after the ACK read returned (a late/duplicate frame in a later segment, or unsolicited bytes while the connection sat idle ‚Äî they buffer in the `StreamReader` where the in-transaction flag cannot see them, and also mask a subsequent FIN from `at_eof()`) veto reuse via the ¬ß2.3 liveness check: close + redial, uncharged.
3. **Reuse-time liveness check + the one sanctioned internal reconnect (reconnect-before-first-byte).** Before writing on a cached connection, check cheaply ‚Äî no I/O round-trip ‚Äî `writer.is_closing()`, buffered unsolicited bytes (the reuse-time half of the ¬ß2.2 desync guard), `reader.at_eof()` (sound for MLLP: a peer sends nothing unsolicited between transactions, so a readable EOF while idle means the peer closed ‚Äî the expected partner-side idle-close), idle > `idle_timeout_seconds`, age > `max_connection_age_seconds`. Any hit ‚Üí close the stale socket (bounded), dial a fresh one, and proceed with the **same** `send()` call. This is **not a message retry** ‚Äî zero payload bytes were written ‚Äî so it is **not charged** to the message (attempts unchanged, no `mark_failed`, no lane-health flip). **Exactly one reconnect attempt per send**: if the fresh dial fails, raise `DeliveryError` (charged, `_describe_error`) ‚Äî never loop.
4. **At-least-once duplicate semantics ‚Äî spelled out.** Once `write()` has been called, a failure (reset during `drain()`, peer close before ACK, ACK timeout) is **indeterminate**: the peer may have received and processed the message even though we saw no ACK. The retry (`mark_failed` ‚Üí backoff ‚Üí resend) may therefore **duplicate**. This is *today's* semantics too (`mllp.py:384-387`, CONNECTIONS.md:232-234 ‚Äî the ACK-lost case) ‚Äî connection reuse does not create it, but a **stale/half-open reused connection makes it more frequent** (a write onto a dead conn often "succeeds" into the TCP buffer and the failure surfaces only at drain/read). Mitigation split:
   - **Failed before any bytes written** (reuse-time check in ¬ß2.3, dial failure) ‚Üí safe: a retry provably cannot duplicate. Handled internally (reconnect-before-first-byte) or charged with a `DeliveryError` whose detail says the connection phase failed.
   - **Indeterminate** (any failure after `write()`) ‚Üí charged `DeliveryError`, normal at-least-once retry; the error detail names the phase (drain/ACK-read) so an operator can distinguish.
   - The **governing invariant is unchanged and stays with the receiver**: "outbound connections must be idempotent" (CLAUDE.md ¬ß2). No internal silent resend loop exists in either class.
5. **Serialized use, asserted.** The worker-per-lane structure already serializes `send()`; the connector **asserts** it rather than trusting it: a `self._sending` guard raises on re-entrant/concurrent `send()` on one instance (a pipeline-invariant bug ‚Äî fail loud; it lands in the worker's internal-error path, `wiring_runner.py:1894-1931`). No lock ‚Äî a lock would silently mask the invariant violation.
6. **Clean shutdown / reload.** `aclose()` closes the cached socket (bounded `wait_closed`, the #55 pattern) and is idempotent + safe concurrently with an in-flight `send()` (the send fails loud ‚Üí charged ‚Üí retried ‚Äî exactly the documented reload race, `wiring_runner.py:1826-1828`). Engine stop (`:1130-1131`) and reload reconcile (`:1295`, `:1305`, `:1319`) already call it, so a settings change swaps the connector and the old socket dies with the old instance ‚Äî **no leak**. No background idle-reaper task: idle expiry is enforced lazily at next send (¬ß2.3), keeping `DestinationConnector` task-free; the cost is one held fd per idle persistent outbound, bounded by outbound count.
7. **TLS.** Reconnect performs a **full re-handshake** through the same prebuilt `SSLContext` (`mllp.py:412`) with the same SNI/verify/mTLS posture (`:450-453`) ‚Äî verification applies on *every* reconnect. TLS session resumption (the context's ticket cache) is an optimization, not a requirement.
8. **Observability.** Every stale-detect + reconnect is logged (INFO for a clean idle-close reconnect with the detected reason, e.g. "peer closed while idle (EOF)"; WARNING when detection followed an error, with `_describe_error` detail ‚Äî metadata only, never payload). The connector keeps an in-memory `reconnects` counter, **log-only in v1**: there is no per-connection counter surface today (`StatsResponse` is engine-wide ‚Äî `api/models.py:323-344`; `GET /connections/{name}/metadata` shows settings/state, not counters) ‚Äî surfacing it is a follow-up (¬ß8). The #46 lane-health events are untouched: an uncharged reconnect-before-first-byte is *not* a `connection_lost` (no delivery failed); a charged `DeliveryError` still edge-triggers it via `_note_lane_unhealthy` (`wiring_runner.py:1892-1893`).
9. **No event-loop blocking** ‚Äî asyncio streams throughout, exactly as today; all closes bounded.

## 3. Acceptance Criteria (EARS)

> Tests land in the implementation PR as `tests/test_mllp_persistent.py` (loopback `asyncio.start_server` ACK peers, as the existing MLLP tests do).

- **AC-1** ‚Äî WHEN a `persistent=True` outbound delivers N messages to a healthy peer, THE ENGINE SHALL use exactly **one** TCP connection for all N (peer sees one accept), established lazily on the first send (no connection at connector build).
  ‚Üí `tests/test_mllp_persistent.py::test_reuses_single_connection_across_sends`, `::test_no_connection_until_first_send`
- **AC-2** ‚Äî WHEN `persistent=False`, THE ENGINE SHALL open and close one connection per delivery, matching the legacy connect-per-send wire behavior (existing `test_mllp_destination` suite green under the flag), with exactly the two intentional hardening deltas named in ¬ß2.1 (bounded close per #55; the AC-8 serial-`send()` assert applies in both modes).
  ‚Üí `tests/test_mllp_persistent.py::test_opt_out_opens_connection_per_send`
- **AC-3** ‚Äî WHEN the partner idle-closes the cached connection and a later send occurs, THE ENGINE SHALL detect the dead connection **before writing any payload bytes**, transparently reconnect once, deliver successfully, and SHALL NOT charge the message (no `mark_failed`, attempts unchanged, no `connection_lost` event).
  ‚Üí `tests/test_mllp_persistent.py::test_peer_idle_close_reconnects_before_first_byte`, `::test_reconnect_before_first_byte_not_charged`
- **AC-4** ‚Äî IF the reconnect dial itself fails, THEN THE ENGINE SHALL raise `DeliveryError` (charged, normal retry policy) whose message carries `_describe_error` detail (type + errno/winerror), and SHALL NOT attempt a second internal reconnect.
  ‚Üí `tests/test_mllp_persistent.py::test_reconnect_failure_charged_with_describe_error_detail`, `::test_exactly_one_internal_reconnect_per_send`
- **AC-5** ‚Äî IF a failure occurs **after** payload bytes were written (reset during drain, peer close before ACK, ACK timeout), THEN THE ENGINE SHALL close + discard the connection, raise `DeliveryError` naming the failing phase (charged; retry per policy ‚Äî the at-least-once duplicate window), and the **next** send SHALL dial fresh.
  ‚Üí `tests/test_mllp_persistent.py::test_midsend_failure_charged_no_internal_retry`, `::test_failed_connection_discarded_next_send_redials`
- **AC-6** ‚Äî WHILE a cached connection has been idle longer than `idle_timeout_seconds` (or is older than `max_connection_age_seconds`), THE ENGINE SHALL NOT reuse it: the next send closes it and dials fresh (uncharged, per AC-3).
  ‚Üí `tests/test_mllp_persistent.py::test_idle_timeout_expires_connection`, `::test_max_age_recycles_connection`
- **AC-7** ‚Äî WHEN the engine stops or a reload replaces/removes the outbound, THE ENGINE SHALL close the cached socket via `aclose()` (bounded `wait_closed`), leaking no socket from the old connector; an in-flight send racing the close SHALL fail loud (charged, retried), never hang.
  ‚Üí `tests/test_mllp_persistent.py::test_aclose_closes_cached_socket`, `::test_reload_swap_closes_old_socket_no_leak`, `::test_aclose_racing_inflight_send_fails_loud`
- **AC-8** ‚Äî IF `send()` is invoked concurrently on one connector instance, THEN THE ENGINE SHALL raise (fail-loud invariant assert), never interleave two frames on one socket.
  ‚Üí `tests/test_mllp_persistent.py::test_concurrent_send_raises`
- **AC-9** ‚Äî WHERE `tls=true`, THE ENGINE SHALL perform a full TLS handshake with unchanged verification (CA, hostname, mTLS) on every reconnect of a persistent connection.
  ‚Üí `tests/test_mllp_persistent.py::test_tls_reconnect_rehandshakes_and_verifies`
- **AC-10** ‚Äî WHEN the partner returns a NAK on a persistent connection, THE ENGINE SHALL preserve `NegativeAckError` semantics unchanged (AR/CR dead-letter, AE retry) AND SHALL keep the connection cached (a NAK is a complete transaction on a healthy transport).
  ‚Üí `tests/test_mllp_persistent.py::test_nak_semantics_unchanged_connection_retained`
- **AC-11** ‚Äî WHEN a reconnect (either class) occurs, THE ENGINE SHALL log it with the detected reason / `_describe_error` detail ‚Äî metadata only, never payload bytes ‚Äî and increment the connector's reconnect counter.
  ‚Üí `tests/test_mllp_persistent.py::test_reconnect_logged_with_detail_no_payload`
- **AC-12** ‚Äî WHILE persistent is on, THE ENGINE SHALL leave `capture_response`/`reingress_to`, `encoding_characters`, `simulate`, and per-lane FIFO order observably unchanged (representative suites re-run with `persistent=True` set explicitly).
  ‚Üí the existing MLLP suites (`test_transports`/`test_mllp_tls`/`test_response_capture`/`test_mllp_encoding_override`/`test_ed_documents_e2e`) run under the shipped default `persistent=False` (this release); representative tests are additionally parametrized over the flag ‚Äî including the explicit `persistent=True` mode ‚Äî rather than wholesale-parametrizing every suite (runtime): round-trip/NAK/connect-failure in `tests/test_transports.py`, the TLS round-trip in `tests/test_mllp_tls.py`, the on-the-wire encoding-override sends in `tests/test_mllp_encoding_override.py`. `tests/test_fifo_ordering.py` exercises store-level lane FIFO with non-MLLP fakes (the flag does not apply); the cross-reconnect order guarantee is `tests/test_mllp_persistent.py::test_fifo_order_preserved_across_reconnect`

## 4. Options considered

1. **Persistent single connection per destination, lazy + reconnect-before-first-byte, opt-out flag. CHOSEN.** Matches the one-serial-consumer-per-lane structure (a bigger unit is wasted); removes per-message handshake + `TIME_WAIT` (the measured WS-C failure mode); the only new duplicate-risk window (stale-conn write) is bounded by the before-first-byte check and governed by the existing idempotency invariant; `aclose()`/reload lifecycle already exists.
2. **Status quo + host tuning only** (docs/SERVICE.md:303-329). Rejected as the end state: host-wide, per-box, operator-dependent, and only *widens* the port budget ‚Äî at higher rates or more destinations it exhausts again; it also leaves 1 RTT + handshake (+ TLS) in every delivery's serial chain. Kept as documented mitigation for `persistent=false` lanes.
3. **Connection pool (N per destination).** Rejected for v1: the delivery worker is the lane's single serial consumer, so >1 connection can never be used concurrently ‚Äî pure complexity (checkout, per-conn liveness, ordering hazards if ever parallelized). Revisit only if delivery ever becomes concurrent per lane (it must not ‚Äî FIFO).
4. **HTTP-style keep-alive probing (background heartbeat / periodic no-op probe).** Rejected: MLLP has no standard no-op frame (an empty frame or stray bytes can confuse real receivers); a background prober adds a task + lifecycle to a task-free connector class and *still* can't close the indeterminate-write window. The lazy at-eof/is-closing check at reuse gives most of the benefit for none of the protocol risk; `poll`-style socket probing is redundant with it.

## 5. Consequences

**Positive** (**realized only when an outbound opts in with `persistent=true` this release; the default posture is unchanged**) ‚Äî per-message connect/close (and its `TIME_WAIT`) eliminated: port pressure drops from per-message to per-reconnect, retiring the WinError 10048/10055 dead-letter mode the 2026-07-02 bench hit (~18.6k dead-letters); 1 RTT + TCP (and TLS) handshake removed from every delivery's serial per-lane chain; partner-side accept-log noise drops. Because `persistent=false` remains the default this release, SERVICE.md host tuning **still applies to the default posture** (connect-per-message) and is not demoted; it becomes an opt-out-only note only once the default flips (¬ß8).

**Negative / risks** ‚Äî a new (small) class of state in the connector, exercised only when `persistent=true` is set: liveness detection is best-effort (see ¬ß7); the indeterminate-write duplicate window becomes *more frequent* (bounded by idle_timeout and the before-first-byte check; governed by receiver idempotency ‚Äî unchanged invariant); one held fd per idle persistent outbound (bounded by outbound count). The default-`false` release keeps existing deployments byte-for-byte on today's wire behavior ‚Äî **no behavior change ships this release**; the observable change (one long-lived connection instead of thousands/day) arrives only for outbounds that opt in, or at the planned default flip (¬ß8, ¬ß7 "Default flip visibility"). Some legacy receivers require connection-per-message and stay on the default.

**Out of scope** ‚Äî inbound MLLP (a listener is persistent by nature ‚Äî untouched); pooled N-connections-per-destination (¬ß4.3); any MLLP-over-TLS policy change (WP-13b posture unchanged); `ack_after=delivered` (still planned, not built); the `Tcp()`/`X12()` destinations, which share the dial-per-send shape (`tcp.py:126`, `x12.py:95`) ‚Äî same pattern can follow in a separate increment; `/stats` schema changes (¬ß8).

## 6. Reliability-invariant checklist

- [x] **At-least-once** ‚Äî every charged failure still goes `DeliveryError` ‚Üí `mark_failed` ‚Üí backoff ‚Üí retry (or `NegativeAckError` policy); the only internal reconnect provably wrote zero payload bytes; no internal resend loop can double-send; a crash mid-send leaves the row in-flight for `reset_stale_inflight` exactly as today.
- [x] **Idempotent-receiver invariant governs duplicates** ‚Äî the indeterminate-write window is the existing ACK-lost window (CONNECTIONS.md:232-234), made less rare, not new; documented in CONNECTIONS.md as part of this change.
- [x] **Per-lane FIFO unaffected** ‚Äî same single worker, same serialized sends, same claim path (`claim_next_fifo` untouched); a reconnect happens inside one send, never reorders.
- [x] **Count-and-log** ‚Äî no message is dropped or newly accepted anywhere; failures land in the same `mark_failed`/`dead_letter_now` paths with richer (`_describe_error`) detail.
- [x] **ACK-on-receipt untouched** ‚Äî this ADR is outbound-only; the inbound listener, `build_ack`, and NAK-synchronous semantics are not modified.
- [x] **Finalizer sole disposition authority** ‚Äî untouched; only transport internals change.
- [x] **No event-loop blocking** ‚Äî asyncio streams; every `wait_closed` bounded (#55 pattern).
- [x] **Independent lanes** ‚Äî one connection per destination instance; a slow/dead partner still affects only its own lane.
- [x] **PHI** ‚Äî reconnect logs carry socket/OS metadata (`_describe_error`) only, never frame bytes; no new durable storage.
- [x] **TLS posture** ‚Äî full verification on every reconnect; no downgrade path; `tls_verify=false` escape hatch semantics unchanged.

## 7. Residual risks

- **Liveness detection is best-effort.** `at_eof()`/`is_closing()` see a peer FIN, not a silently dead path (NAT drop, partner power loss): that case writes into the void and surfaces as ACK-timeout ‚Üí charged `DeliveryError` ‚Üí retry ‚Äî exactly today's behavior for a mid-send partner death. `idle_timeout_seconds` bounds how stale a reused connection can be.
- **Partner devices that misbehave on reuse** (accept the connection, process only the first frame). Mitigation: `persistent=false` per outbound; document in CONNECTIONS.md which class of partner needs it.
- **ADR 0066 interaction.** Pooled per-stage claimers change *who claims*, not who sends ‚Äî but the implementation must re-verify that the pooled dispatcher preserves one-serial-sender-per-destination before both land; the AC-8 concurrent-send assert is the tripwire.
- **Reload race** (in-flight send vs `aclose()`) ‚Äî pre-existing, documented, at-most-one racing send fails-and-retries (`wiring_runner.py:1826-1828`); the persistent socket adds no new window, but the assert + bounded close must keep it hang-free on the Windows Proactor.
- **Decoder desync on reuse** ‚Äî a peer that sends extra bytes beyond its one ACK (a duplicate/late frame) would corrupt the next transaction's framing; guarded twice (¬ß2.2): close-on-leftover-bytes within the transaction, plus the reuse-time buffered-bytes veto for frames that arrive in a later segment or while the connection sits idle. Both halves cost a reconnect, never a mis-parse. **Residual window:** a stray frame that lands *mid-transaction* ‚Äî after this send's payload write, before its ACK read completes ‚Äî is indistinguishable from the reply and is parsed as this send's ACK; closing that requires strict MSA-2‚ÜîMSH-10 response correlation, which is deliberately demand-gated (docs/BACKLOG.md #82 ‚Äî some real peers echo empty/wrong MSA-2, so it must be a per-connection opt-in when a partner needs it). A peer known to emit unsolicited or duplicate frames should set `persistent=false` until then.
- **Default flip visibility** ‚Äî this release ships `persistent=false` by default, so partners see **no change** unless an outbound opts in. When the default flips (¬ß8) ‚Äî or an operator opts in earlier ‚Äî partners monitoring connection counts will see one long-lived connection instead of thousands/day. Low risk (standard MLLP), but release-note it at the flip.
