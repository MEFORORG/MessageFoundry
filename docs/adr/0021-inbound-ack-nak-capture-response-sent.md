# ADR 0021 — Inbound ACK/NAK capture ("Response Sent"), an increment of ADR 0013

- **Status:** **Accepted (2026-06-19, owner go).** No code yet (`eventlog` #16, Lane S, stays on-trigger). Conceptually "ADR 0013 Increment 3" (the inbound twin of 0013's outbound `response` capture), but it is a new decision and so carries its own number, **0021**. (`0019` and `0020` are taken — 0019 = the staged KeyProvider ADR; 0020 = protocol-diagnostic capture.) **Ratification status (2026-06-19):** §§1-6 (ACK capture) + the new **§7** (metadata-only connection-error event log) are drafted and adversarially verified; the one design defect (an un-capturable TLS-accept event) is resolved by scoping it out of v1 (see §7). **Accepted with the two v1 scope knobs resolved (owner go, 2026-06-19): (A) §7 ships store-row-only** — the optional `AlertSink.connection_error` wiring is a fast-follow once operators want it; **(B) v1 is capture + store + purge only** — the API/console read surface is deferred with its consumer (§5 posture).
- **Built:** nothing yet. This document is the design only.
- **Decision in one line:** record the ACK/NAK MessageFoundry **returns** to an inbound sender as a queryable, encrypted, audited, retention-purged artifact correlated to the inbound message — by **extending the existing ADR 0013 `response` table** with a `kind` discriminator plus two cheap plaintext typed columns (`ack_code`, `ack_phase`), reusing the entire 0013 cipher/purge/correlate/API/audit machinery, with capture done **synchronously** in `_handle_inbound` so there is no fire-and-forget vs key-rotation race. A companion **§7** adds a metadata-only `connection_event` log for **pre-ingress** listener failures (bad framing, peer reset, allowlist/capacity refuse, oversize frame) that never get a `message_id` — a small new id-keyed table (NOT `message_events`, whose `message_id` is `NOT NULL`), captured via a fire-and-forget engine callback the source connector calls (no `transports/`→`store/` import), OFF by default per-inbound (`capture_connection_errors`).
- **Related:** [ADR 0013](0013-query-response-orchestration.md) (the `response` artifact table + `complete_with_response`/`correlate_response`/`purge_message_bodies` machinery this reuses), [ADR 0001](0001-staged-pipeline-architecture.md) (ACK-on-receipt + the listener path this hooks), [ADR 0004](0004-payload-agnostic-ingress.md) (non-HL7 inbounds return None / no ACK — nothing to capture), [CLAUDE.md](../../CLAUDE.md) §2/§9, [ADR 0020](0020-protocol-diagnostic-capture.md) (the raw-frame `protocol_trace` tier §7 deliberately does **not** build — §7 is metadata-only), [alerts.py](../../messagefoundry/pipeline/alerts.py) (the `AlertSink` §7.6 optionally reuses).

## Context

ADR 0013 captures the partner's reply to an **outbound** delivery. The symmetric gap is the **inbound** ACK/NAK MessageFoundry itself **generates and returns** to a sender — Corepoint's "Response Sent". `build_ack` builds the ACK string ([mllp.py:144-204](../../messagefoundry/transports/mllp.py)) and is called from **four** sites in `RegistryRunner._handle_inbound`: AR-decode ([wiring_runner.py:705](../../messagefoundry/pipeline/wiring_runner.py)), AR-parse ([735](../../messagefoundry/pipeline/wiring_runner.py)), AE-strict ([755](../../messagefoundry/pipeline/wiring_runner.py)), and AA-success ([776](../../messagefoundry/pipeline/wiring_runner.py)). `record_received`/`enqueue_ingress` already **return** the `message_id` ([store.py:1652, 1290](../../messagefoundry/store/store.py)) — but every site **discards** it. Nothing records the ACK we send; there is no store column, audit event, or test for an inbound ACK today (repo-wide grep for `record_ack`/`ack_sent`/`response_sent` = 0 hits). `_handle_inbound` is the **only** scope holding BOTH the store-returned `message_id` AND the `build_ack` string, and `build_ack` is already imported into the runner ([wiring_runner.py:70](../../messagefoundry/pipeline/wiring_runner.py)) — so capture is engine-side, with no `transports/` change and no `InboundHandler` contract change.

The sensitivity: the AA ACK echoes MSH (control fields — PHI), and an AE/AR NAK frame quotes the **offending field VALUE** from hl7apy's error string (the very PHI the existing #120 invariant excludes from durable storage — the code at [wiring_runner.py:750-758](../../messagefoundry/pipeline/wiring_runner.py) deliberately persists only `safe_text(joined)` and keeps the fuller AE frame transient).

## Decision

### §1 Extend `response`, do not add a new table

The `response` table is already immutable, finalizer-invisible (`_maybe_finalize_message` scans `FROM queue` only, verified [store.py:3583-3584](../../messagefoundry/store/store.py)), composite-PK cipher-passed (`_encrypt_existing_rows`/`reencrypt_to_active` over `body`+`detail`), and retention-purged on all three backends ([store.py:3342-3346](../../messagefoundry/store/store.py) + PG/SQL-Server twins). Reusing it (vs a new `inbound_ack` table) avoids ~2x per-backend DDL/cipher/purge duplication and **guarantees ACK PHI is encrypted at rest on SQL Server** (where `response.body`/`detail` are ciphertext — [sqlserver.py:1050-1051](../../messagefoundry/store/sqlserver.py) — but `message_events.detail` is plaintext).

### §2 Schema deltas (all three backends)

Three new columns on `response`: `kind` ('response' = existing outbound reply, the DEFAULT | 'ack_sent' = inbound ACK), `ack_code` ('AA'|'AE'|'AR'|'CA'|'CE'|'CR'; NULL for 'response'; non-PHI disposition metadata, plaintext), `ack_phase` ('decode'|'parse'|'strict'|'ingest'; NULL for 'response'; plaintext). Column reuse for an `ack_sent` row:
- `destination_name` = a **sentinel-prefixed synthetic name** `"\x1fack:" + ic.name`. The leading `\x1f` (unit separator, already the artifact_ref delimiter at [store.py:2152](../../messagefoundry/store/store.py)) cannot occur in a real connection name (names follow `[TYPE]_[PARTNER]_[MESSAGE]`, identifier-validated), so the ack row's `(message_id, destination_name)` key is **provably disjoint** from every outbound destination's key even if an operator names an inbound identically to an outbound.
- `response_seq` = `1 + COALESCE(MAX(response_seq),0) WHERE message_id=? AND destination_name=? AND kind='ack_sent'` — the **`kind` predicate is mandatory** in the seq SELECT. One ACK per received message in practice → seq=1. Append-only, replay-stable by the same argument as ADR 0013.
- `body` = **only for `ack_code='AA'`** (the success ACK echoes MSH but quotes no rejected field value) → `self._enc(...)`. For **every NAK** (AE/AR/CE/CR) `body = NULL`.
- `outcome` = 'accepted' (AA/CA) | 'rejected' (AE/CE/AR/CR).
- `detail` = a **mandatory `safe_text`-scrubbed**, bounded reason for a NAK (e.g. `safe_text(reason)[:200]`) → `self._enc(...)`; NULL for AA.
- `captured_at` = ACK generation time.

**PHI rationale for the AA-body/NAK-NULL split:** persisting an AE/AR frame would re-introduce exactly the PHI-quoting field value the #120 invariant excludes from durable storage. So for NAKs we persist only the scrubbed reason + code/phase — which IS the reconciliation artifact ("what code at 14:32, and why AR?"). The AA frame quotes no rejected value (it echoes MSH/MSA control fields), so storing it (encrypted) matches ADR 0013's outbound-body semantics.

Per-backend DDL + migration (no pre-existing `response` migration block — authored from scratch): **SQLite** `_SCHEMA` CREATE + a NEW `_migrate` guarded block (`PRAGMA table_info(response)` then `ALTER TABLE response ADD COLUMN` for the three; static literal names; existing rows backfill `kind='response'` via the DEFAULT). **Postgres** `_SCHEMA` CREATE + a NEW response branch inside `_migrate_lease_columns` (the information_schema-gated migrator — do NOT use a bare per-open `ADD COLUMN IF NOT EXISTS`, which takes ACCESS EXCLUSIVE every open). **SQL Server** `_SCHEMA` inline CREATE + guarded `IF COL_LENGTH('response','kind') IS NULL ALTER TABLE response ADD kind NVARCHAR(32) NOT NULL CONSTRAINT df_response_kind DEFAULT 'response'` (+ ack_code/ack_phase NULLable) — adding a NOT NULL column with a **constant** default is metadata-only (no rewrite) on SQL Server 2012+/2016+ (covers the CI 2022 container); **a migration test must time this ALTER on a pre-populated table**, with a documented fallback (NULLable add → backfill → set NOT NULL in batched steps) if any rewrite is observed. **NO `ix_response_ack` index in v1** on any backend (its only consumer — fleet ack-code analytics — is deferred; a filtered index is a SET-options foot-gun on the `response` INSERT hot path for zero v1 benefit). `CapturedResponse` ([store.py:133-145](../../messagefoundry/store/store.py)) gains `kind: str = 'response'`, `ack_code: str | None = None`, `ack_phase: str | None = None` (defaults keep outbound construction sites unaffected).

### §3 Capture hook (`_handle_inbound`, synchronous)

New store method `record_ack_sent(*, message_id, inbound_name, ack_body, ack_code, ack_phase, outcome, detail=None, now=None)` on the Store protocol + all 3 backends: synthetic `dest = "\x1fack:" + inbound_name`; under the backend's single-transaction discipline (the `complete_with_response` pattern — explicit BEGIN/single commit/rollback under `self._lock` on SQLite, `conn.transaction()` on PG, one cursor/commit with a leading SELECT to open the txn on SQL Server) it does `SELECT 1+COALESCE(MAX(response_seq),0) ... AND kind='ack_sent'` then `INSERT INTO response (... body=enc(ack_body), outcome, detail=enc(detail), ..., kind='ack_sent', ack_code, ack_phase)`. It touches **NO `queue` row**, calls **NO `_maybe_finalize_message`**, emits **NO `delivered` event**. SQL Server inlines the empty-guard exactly as `complete_with_response` ([sqlserver.py:1050-1051](../../messagefoundry/store/sqlserver.py)): `enc_body = self._cipher.encrypt(ack_body) if ack_body else ack_body` (there is no `self._enc` on that backend).

Wiring in `_handle_inbound` (un-discard the `message_id`, capture SYNCHRONOUSLY before returning the ACK): at each of the four ACK-emitting sites capture the returned id and, if `ic.capture_ack`, `await self._capture_ack(...)` before returning the ACK string. AR-decode → `ack_body=None, code='AR', phase='decode', detail=safe_text('decode error')`; AR-parse → `ack_body=None, code='AR', phase='parse', detail=safe_text(str(exc))[:200]`; AE-strict → widen `_record` ([wiring_runner.py:778](../../messagefoundry/pipeline/wiring_runner.py), single caller) to RETURN the id, then `ack_body=None, code='AE', phase='strict', detail=persisted[:200]` (the **already-scrubbed** `f"strict-validation failed: {safe_text(joined)}"`, NOT `joined` — the unscrubbed AE frame is NEVER persisted); AA-success → `ack_body=ack, code='AA', phase='ingest', outcome='accepted', detail=None`. No-ACK paths (`AckMode.NONE`; non-HL7 at [710-723](../../messagefoundry/pipeline/wiring_runner.py); File source) capture NOTHING (the table records only ACKs actually generated — exactly Corepoint's "Response Sent" semantics).

`_capture_ack` is **fail-soft on the store write**: the INSERT is `await`ed (transactionally adjacent — no scheduling-delay window vs rotation) but a store failure is **swallowed + logged WARNING** (`safe_exc`, never the body), so a capture error can never flip an AA to a NAK or tear down the MLLP per-connection loop ([mllp.py:615](../../messagefoundry/transports/mllp.py)). **Why synchronous (rotation-race fix):** a deferred `create_task` capture could land under the OLD key in a region `reencrypt_to_active` already scanned past, defeating rotation for that row. A synchronous, lock-serialized INSERT is naturally serialized against the rotation pass exactly like `complete_with_response` — written under whatever key is active and reachable by active-write + retired-key-retention. The `create_task`, inflight cap, and "best-effort" loss window are removed; the only loss window is a swallowed store-write failure (logged WARNING).

### §4 Config (per-inbound, OFF by default)

First-class field on `InboundConnection` ([wiring.py:1244-1262](../../messagefoundry/config/wiring.py)) — `capture_ack: bool = False` — NOT a transport-settings entry, because the RegistryRunner (pipeline) does the capture, so it belongs with `content_type`/`ack_after` (runner-honored fields). The ADR documents that this is **not** a structural mirror of `capture_response` (which lives in `spec.settings` on the outbound); it is the inbound analogue. Thread through the `inbound()` factory ([wiring.py:1453](../../messagefoundry/config/wiring.py)) and `build_inbound_connection` ([wiring.py:1435](../../messagefoundry/config/wiring.py), the shared code-first + connections.toml choke point): guards (raise `WiringError`, caught by `messagefoundry check`/dry-run) — `capture_ack=True` requires `ack_mode != AckMode.NONE` and `content_type is ContentType.HL7V2`. Add `capture_ack` to `_INBOUND_KEYS` and decode via `_require_bool(...)` defaulting False (a top-level connection-table key, not a `[settings]` key). No row-count cap needed (one row per received HL7 message, naturally bounded by inbound volume + retention).

**NEW inbound-side runner-start fail-closed guard.** The existing outbound guard ([wiring_runner.py:386-395](../../messagefoundry/pipeline/wiring_runner.py)) loops `registry.outbound` and keys on a destination attribute, so it structurally cannot see `ic.capture_ack`. Add a SEPARATE inbound loop at runner start (near [wiring_runner.py:404](../../messagefoundry/pipeline/wiring_runner.py)): for each `ic` with `capture_ack`, raise if `not supports_response_capture`, and re-validate the wiring guards (`ack_mode != NONE`, `content_type is HL7V2`) so a hand-edited `connections.toml` that bypassed `build_inbound_connection` still fails closed.

### §5 API + RBAC + audit (reuse the existing route)

Reuse `GET /messages/{message_id}/responses` ([app.py:1094-1143](../../messagefoundry/api/app.py)) — NO new endpoint, NO new permission. `correlate_response` ([store.py:2162-2184](../../messagefoundry/store/store.py)) now also returns `ack_sent` rows: add `kind, ack_code, ack_phase` to the SELECT projection ([store.py:2168](../../messagefoundry/store/store.py)) and the `CapturedResponse` construction ([store.py:2174](../../messagefoundry/store/store.py)). The `ORDER BY destination_name, response_seq` stays — ack rows sort under their `\x1fack:` synthetic name, distinct from every real destination, so the "latest seq per destination is the authoritative reply" semantics for outbound rows are **provably unaffected**. The route already gates `require_phi_read(Permission.MESSAGES_READ)`, 404s out-of-channel-scope with `_audit_channel_denied`, splits `body` on `MESSAGES_VIEW_RAW`, runs `redact_unauthorized` (nulls `detail` for callers lacking `MESSAGES_VIEW_SUMMARY`, [field_authz.py:67-68](../../messagefoundry/api/field_authz.py)), and emits `response.read` audit + `record_view` when bodies exposed — ack rows ride ALL of it unchanged. `CapturedResponseInfo` ([api/models.py:72-82](../../messagefoundry/api/models.py)) gains `kind: str`, `ack_code: str | None`, `ack_phase: str | None` — non-PHI disposition metadata, NO new `field_authz` entry (PHI `body` stays on the view_raw split; `detail` stays on the view_summary `field_authz` gate). The console "Response Sent" view consumes the same endpoint filtered to `kind=='ack_sent'`. Fleet ack-code analytics (`GET /responses?kind=ack_sent&ack_code=AR`) + its index are **deferred** to ship with their consumer.

### §6 PHI: encryption + retention (ZERO new cipher/purge code)

`ack_body` (AA only) → `self._enc(...)` into `response.body`; scrubbed `detail` → `self._enc(...)` into `response.detail` on SQLite+PG via the SAME `_enc` chokepoint; SQL Server inlines `self._cipher.encrypt(x) if x else x`. `kind`/`ack_code`/`ack_phase` are non-PHI plaintext. Cipher migration + rotation: `_encrypt_existing_rows` and `reencrypt_to_active` already iterate `response` body+detail by composite PK — ack rows are just more rows, ZERO change; the synchronous-capture decision keeps every ack row reachable by the rotation pass. Retention: `purge_message_bodies` already nulls `response.body`+`detail` in place for the eligible set ([store.py:3342-3346](../../messagefoundry/store/store.py) + PG/SQL-Server twins) — ack rows purge identically (NAK rows already have `body=NULL`); metadata (kind/ack_code/ack_phase/outcome/captured_at) survives for audit. **No RetentionPass change.**

### §7 Metadata-only connection-error event log (pre-ingress, no `message_id`)

§§1-6 capture the ACK we **generate** for a message that at least *parsed* — every one of those rows keys on a real `message_id` returned by `record_received`/`enqueue_ingress`. The genuine gap they **cannot** cover is the **pre-message failure**: a connection event that fails *before* any ingress row exists, so there is no `message_id` to correlate, no `build_ack` string, and `_handle_inbound` **never runs**. These live entirely inside the MLLP listener loop (`MLLPSource._on_client`, [mllp.py:575-640](../../messagefoundry/transports/mllp.py)) and are dropped to a WARNING log today:

- **allowlist / IP refuse** — `peer_ip_allowed` rejects the peer ([mllp.py:584-590](../../messagefoundry/transports/mllp.py)).
- **at-capacity refuse** — `self._active >= self.max_connections` ([mllp.py:591-592](../../messagefoundry/transports/mllp.py)).
- **oversize-frame reject** — `MLLPFrameError` over `max_frame_bytes` ([mllp.py:612-617](../../messagefoundry/transports/mllp.py)).
- **peer reset mid-frame** — the bare `OSError` paths ([mllp.py:618-619](../../messagefoundry/transports/mllp.py) / [628-629](../../messagefoundry/transports/mllp.py)).
- **bad/garbage framing + last-resort codec error** — the catch-all at [mllp.py:620-627](../../messagefoundry/transports/mllp.py).

This is **not** an ADR-0020 raw-frame `protocol_trace` (that tier is **dropped** by the Plan-3 §G cull): we capture **metadata only** — never a raw frame, a message body, or an HL7 field value.

**Out of v1 scope — pre-callback TLS handshake failures.** When an inbound terminates TLS, asyncio runs the handshake at the transport layer the moment `ssl=` is passed to `asyncio.start_server` ([mllp.py:538-540](../../messagefoundry/transports/mllp.py)); a handshake rejection (bad mTLS client cert, protocol mismatch) means `_on_client` is **never invoked**, so there is no per-connection seam to attribute it to `ic.name` or to the capture callback. Such failures surface only via asyncio's loop exception handler ([last_resort.py:39](../../messagefoundry/last_resort.py), `set_exception_handler`) and stay there in v1. A future increment may add `tls_accept_failed` **only** with a proven seam (e.g. a protocol-factory wrapper observing `connection_lost(exc)` / `ssl.SSLError` and resolving the listening connection from the local bind) **and** a test that forces an mTLS rejection and asserts the row — it is excluded from the v1 `kind` enum until then.

#### §7.1 Home: a NEW `connection_event` table, NOT `message_events`

These rows **cannot** live in `message_events`. Its `message_id` column is `NOT NULL REFERENCES messages(id)` ([store.py:452-459](../../messagefoundry/store/store.py); SQL Server twin [sqlserver.py:114-118](../../messagefoundry/store/sqlserver.py)), and every writer mints a `messages` row first (`record_received` does `_insert_message` then the event INSERT under one txn, [store.py:1728-1748](../../messagefoundry/store/store.py)). A pre-ingress failure has **no** message — synthesizing a placeholder `messages` row to satisfy the FK would (a) inflate received-volume counts (the count-and-log invariant counts *received messages*, and a refused TLS handshake is not a received message), and (b) leak a pre-ingress failure into `_maybe_finalize_message`'s disposition surface. They are also **not** a `response`/`ack_sent` row: that table keys on `(message_id, destination_name, response_seq)` ([store.py schema](../../messagefoundry/store/store.py)) and is reached by `correlate_response` per message-id — there is no id to key on.

So §7 adds a small dedicated table, a sibling of `message_events` but **id-keyed only** (no message FK):

```sql
CREATE TABLE IF NOT EXISTS connection_event (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    ts          REAL    NOT NULL,
    connection  TEXT    NOT NULL,   -- inbound connection name (config metadata; never payload)
    transport   TEXT    NOT NULL,   -- 'mllp' (ConnectorType.value) — which listener
    kind        TEXT    NOT NULL,   -- bounded enum: see §7.3
    peer_host   TEXT,               -- peer IP from getsockname/peername (socket metadata; nullable)
    reason      TEXT                -- safe_text-scrubbed, bounded; plaintext metadata, not PHI
);
CREATE INDEX IF NOT EXISTS ix_connection_event_conn ON connection_event(connection, ts);
```

Backend parity mirrors `message_events`: **SQLite** `_SCHEMA` CREATE + a NEW `_migrate` guarded `CREATE TABLE IF NOT EXISTS` (no `ALTER` — a fresh table, so no column-add timing concern); **Postgres** the same `IF NOT EXISTS` CREATE in `_SCHEMA`; **SQL Server** an `IF OBJECT_ID('connection_event','U') IS NULL CREATE TABLE …` inline, exactly like the `message_events` block at [sqlserver.py:114-118](../../messagefoundry/store/sqlserver.py). A brand-new table needs **no** `ADD COLUMN` migration, so the §7 schema delta carries **none** of §2's NOT-NULL-default rewrite risk.

#### §7.2 Capture seam: an engine-supplied callback, never a store import in `transports/`

The pre-message failures happen in `MLLPSource`, which **must not import `store/`** (CLAUDE.md §4 one-way dependency: `transports/` never imports `store/`/`pipeline/`). The source today only holds an `InboundHandler` (`Callable[[bytes], Awaitable[str | None]]`, [base.py:42-45](../../messagefoundry/transports/base.py)), which fires only on a *complete* message — useless here. So §7 adds a **second, optional, fire-and-forget callback** to `SourceConnector.start`, supplied by the runner, that the engine implements as a thin closure over the store:

```python
# transports/base.py — the new contract, store-agnostic (just a coroutine the source may call):
ConnectionEventSink = Callable[[str, str | None, str | None], Awaitable[None]]
#                                ^kind  ^peer_host    ^reason     (connection + transport bound by the runner)

async def start(self, handler, *, leader_gate=None, on_connection_event=None): ...
```

The runner builds `on_connection_event` per-inbound as `lambda kind, peer, reason: self._capture_connection_event(ic, kind, peer, reason)`, which calls a new `store.record_connection_event(*, connection, transport, kind, peer_host=None, reason=None, now=None)`. The transport stays store-agnostic (it calls an injected coroutine, exactly as `send()` *returns* a value rather than writing the store — the same one-way discipline ADR 0013 used to reject the "capture callback into `send`"). `on_connection_event=None` (the default, and what dry-run / direct callers pass) makes every call site a no-op, so the listener is **byte-identical when off**.

**Fail-soft, never tears down accept (HARD CONSTRAINT 3).** Every call is wrapped so a capture failure can never break the listener: the source guards `if on_connection_event is not None: try: await on_connection_event(...) except Exception: logger.warning(... safe_exc(exc) ...)`. A wedged/slow store write must not block `accept`, so the runner-side `_capture_connection_event` is itself best-effort (swallow + log WARNING on a store error, mirroring `_capture_ack` in §3). The capture is *adjacent to* the existing WARNING log (which stays), so even a swallowed store write leaves the operator the log line.

#### §7.3 The bounded `kind` enum + per-field PHI rationale

`kind` is a **closed engine-defined vocabulary** (an enum, not free text), one value per listener failure class:
`peer_not_allowlisted | at_capacity | frame_oversize | peer_reset | framing_error` (the five v1 values; `tls_accept_failed` is deferred — see *Out of v1 scope* above).
Each call site in `_on_client` maps to exactly one, e.g. [mllp.py:587](../../messagefoundry/transports/mllp.py) → `peer_not_allowlisted`, [mllp.py:612-616](../../messagefoundry/transports/mllp.py) → `frame_oversize`, [mllp.py:620-626](../../messagefoundry/transports/mllp.py) → `framing_error`.

**Why every column is non-PHI:**
- **`connection`** — the inbound connection *name* (`ic.name`), operator-authored config metadata (`[TYPE]_[PARTNER]_[MESSAGE]`), already non-PHI and already in `message_events`/`audit_log`/every AlertSink payload. Never derived from the wire.
- **`transport`** — `ConnectorType.value` (`'mllp'`); a static label.
- **`kind`** — a fixed engine enum value; carries no wire content by construction.
- **`peer_host`** — the peer IP from the **socket** (`writer.get_extra_info('peername')`, the same value already logged at [mllp.py:585-589](../../messagefoundry/transports/mllp.py)) — connection metadata, not message payload. An IP is network metadata, not a HIPAA identifier of a patient; it is already WARNING-logged today. Nullable (a UNIX socket / unresolved peer yields `None`, per `_peer_ip`).
- **`reason`** — a **`safe_text`-scrubbed, bounded** diagnostic (`safe_text(str(exc))[:200]` via [redaction.py:51-61](../../messagefoundry/redaction.py)), the same PHI chokepoint §3 uses. For a pre-message failure the exception is a framing/TLS/IO error (`MLLPFrameError`, `ssl.SSLError`, `OSError`) whose text is transport diagnostics, not field values — but it is scrubbed regardless so a hostile garbage frame that happens to embed HL7-shaped bytes in an error string can never land verbatim. Crucially, the **raw frame is never passed to `record_connection_event`** — only the scrubbed exception text — so there is no body/field-value path at all.

There is deliberately **no message body, no raw frame, no MSH, no field value** in the schema or any call site.

#### §7.4 Config: per-inbound, OFF by default (mirrors §4's `capture_ack`)

A first-class field on `InboundConnection` ([wiring.py:1244-1262](../../messagefoundry/config/wiring.py)) — `capture_connection_errors: bool = False` — beside `capture_ack` (§4) and `content_type`/`ack_after` (runner-honored fields, not `[settings]`), threaded through the `inbound()` factory and `build_inbound_connection` ([wiring.py:1435](../../messagefoundry/config/wiring.py)), added to `_INBOUND_KEYS` ([connections_file.py:78](../../messagefoundry/config/connections_file.py)), decoded via `_require_bool(...)` defaulting `False`. When `False`, the runner passes `on_connection_event=None` and the listener is byte-identical. **Wiring guard:** it is only meaningful on a **listen** source (MLLP/TCP) — on a poll/file source (no peer, no framing) it is a `WiringError` at `check`/dry-run, exactly as `source_ip_allowlist` is gated to listen sources ([wiring.py:1393-1397](../../messagefoundry/config/wiring.py)). Unlike `capture_ack`, it does **not** require `content_type is HL7V2` (a TLS/framing failure is content-agnostic) nor `ack_mode != NONE` (these failures precede any ACK). No row-count cap is needed beyond retention: the rows are bounded by the realistic rate of refused/failed connections and pruned by §7.5; a degenerate connection-flood is already bounded by `max_connections` and the realert throttle on the alert side (§7.6). **Ordering note:** §7 presumes §§1-6 land first — `capture_connection_errors` reuses the same `InboundConnection`/`inbound()`/`build_inbound_connection` + inbound runner-start guard §4 introduces for `capture_ack` (neither `capture_ack` nor that guard exists in the code yet — both are this ADR's own proposals); if §7 ships independently, that scaffolding is built here.

#### §7.5 PHI: encryption-at-rest, retention-purge, audit

- **Encryption at rest.** `reason` is the only column that could in theory carry residual free text. On SQLite/Postgres, add `('connection_event', 'reason')` to `_CIPHER_COLUMNS` ([store.py:838-850](../../messagefoundry/store/store.py)) so it rides the existing `_encrypt_existing_rows`/`reencrypt_to_active` id-keyed loops (the table has an `id` PK, so unlike `response` it needs **no** bespoke cipher pass). **SQL Server: flagged plaintext.** Just like `message_events.detail`, `messages.error`, and `queue.last_error`, this backend keeps the detail column plaintext at rest ([sqlserver.py:196-202](../../messagefoundry/store/sqlserver.py)); `connection_event.reason` joins that documented set. This is acceptable because `reason` is metadata-only and `safe_text`-scrubbed at the chokepoint — the **same** control the existing plaintext SQL-Server detail columns rely on — and the at-rest encryption of that class on SQL Server is the same tracked defense-in-depth follow-up (docs/PHI.md). The other four columns (`connection`/`transport`/`kind`/`peer_host`) are non-PHI metadata and stay plaintext on every backend by design.
- **Retention purge.** Two options, and the ADR picks **age-based delete** over body-null: unlike `messages`/`response`, a `connection_event` row has **no body to null and no FK to keep** (it does not reference `messages`), so it is purged by a plain `DELETE FROM connection_event WHERE ts < :older_than` added inside `purge_message_bodies`' existing transaction ([store.py:3515](../../messagefoundry/store/store.py) + PG/SQL-Server twins), keyed on the same `older_than` cutoff. (A body-null UPDATE would leave an ever-growing metadata table; a delete bounds it.) The row is *purged*, not retained-with-nulled-body, because none of its surviving columns are an audit obligation tied to a `messages` row. These rows are purged by **event age against the same retention cutoff as message bodies** — an intentional reuse of the body-retention window for a metadata clock (not a per-message-eligibility delete), mirroring the existing non-message-keyed `state` purge already run in that transaction on SQL Server ([sqlserver.py:1590](../../messagefoundry/store/sqlserver.py)); connection-error rows are not expected to outlive message bodies.
- **Audit / read surface.** Exposure is **deferred with its consumer** (same posture as §5's deferred fleet ack analytics and ADR 0013's read-surface gating): when a console "Connection Errors" view is built, it reads through a new engine API route gated `require_phi_read(Permission.MESSAGES_READ)` and emitting an audit event, reusing the field-authz/`record_view` machinery. v1 ships the **capture + store + purge** only (engine-side); no new endpoint or permission lands until the view does. This keeps §7 strictly engine-side (no `api/`/`console/` import, CLAUDE.md §4).

#### §7.6 Optional alerting (reuse the AlertSink, no new transport)

A burst of `peer_not_allowlisted` / `framing_error` is operationally interesting (a misconfigured partner, a probe). The `AlertSink` Protocol ([alerts.py:27-56](../../messagefoundry/pipeline/alerts.py)) already carries the right non-PHI shape (connection name + metadata, never a body). §7 **may** add a `connection_error(name, *, kind, detail)` emit method to `AlertSink`/`LoggingAlertSink`/`NotifierAlertSink`, fired from `_capture_connection_event` alongside the store write, keyed `connection_error:<connection>` so `NotifierAlertSink._emit`'s realert throttle ([alert_sinks.py:406-427](../../messagefoundry/pipeline/alert_sinks.py)) collapses a flood to one notification per cooldown. The payload carries only `connection`/`kind`/scrubbed `detail` — the AlertSink's existing "no PHI, connection + shape only" contract holds unchanged. This is **optional and gated by the same `capture_connection_errors` flag**; with it off, no event and no alert. (Whether v1 wires the alert or only the store row is the one genuinely owner-facing knob — see *To resolve*.)

## Consequences

**Positive**
- Corepoint's "Response Sent" becomes a queryable, encrypted, audited, retention-purged artifact correlated to the inbound message, reusing five subsystems of ADR 0013 machinery with **zero new cipher/purge code** and one new store method + three columns.
- `response` is not a queue stage; `_maybe_finalize_message` scans `FROM queue` only → an `ack_sent` row can NEVER pin a message out of `PROCESSED` or flip it to `ERROR`. Capture happens AFTER the ingress/record commit, so persist-before-ACK ordering is untouched.
- Synchronous capture eliminates the fire-and-forget vs key-rotation race entirely and removes the `create_task`/inflight-cap/best-effort-loss machinery.
- **Byte-identical when off** (`capture_ack=False`, the default): the four ACK sites keep discarding the id, no `response` row is written.
- NAK bodies are NOT persisted — the PHI-quoting AE/AR frame stays a transient wire ACK; only the scrubbed reason + code/phase are durable (#120 preserved).
- **§7** closes the pre-ingress blind spot (garbage framing, peer reset, allowlist/capacity refuse, oversize frame) as a queryable, retention-purged, **metadata-only** log without inflating received-volume counts or touching the finalizer — the new `connection_event` table is invisible to both the `messages` count and `_maybe_finalize_message`.

**Negative / costs**
- Synchronous capture adds one local INSERT of ACK latency on the accept path per received HL7 message when `capture_ack` is on (within budget — the ingress commit already dominates; OFF by default).
- **Capture is single-shot at receipt, never replayed.** `_handle_inbound` runs exactly once per physical receipt; `reset_stale_inflight` recovers routed/outbound rows but never re-invokes the listener path. This is correct (the ACK is single-shot on the wire), but it means there is no re-run path that re-derives the ack row — a swallowed store-write failure is the only loss window. (ADR 0013's "on a re-run the reply re-derives" framing does NOT apply here; struck.)
- The seq/PK rule must be keyed on `kind` and ack rows use the `\x1fack:` sentinel `destination_name`, or an inbound name colliding with an outbound destination name could corrupt per-destination authoritative-reply ordering — a deliberate, tested guard.
- Captures the ACK we **GENERATED**, NOT a wire-confirmed drain (the socket write is at [mllp.py:605](../../messagefoundry/transports/mllp.py) after the handler returns) — matches ADR 0013's "what we produced" semantics; a reconciliation user must know captured ≠ delivered-on-wire.
- **§7** adds a second optional `SourceConnector.start` callback (`on_connection_event`) + a new store method + a `connection_event` table on all three backends; on SQL Server its `reason` column is **plaintext at rest** (joining the documented `message_events.detail`/`messages.error` set) — acceptable because it is `safe_text`-scrubbed metadata, never a body or field value.

## Alternatives considered

| Alternative | Why considered | Why rejected | Verdict |
|---|---|---|---|
| **Extend `response` with a `kind` discriminator, capture synchronously** *(chosen)* | Reuses all of ADR 0013's cipher/purge/correlate/API/audit; guarantees ACK PHI encrypted at rest on SQL Server | Three new columns on a hot table; a `_migrate` block per backend | **Adopted** |
| **A new `inbound_ack` table** | Clean separation | ~2x per-backend DDL/cipher/purge duplication; would have to re-prove SQL Server detail-column encryption | **Rejected** |
| **Fire-and-forget `create_task` capture** | Keeps the ACK latency budget untouched | A deferred INSERT can land under the OLD key past the rotation scan, defeating rotation for that PHI row; adds an inflight cap + loss window | **Rejected (→ synchronous)** |
| **Persist the full NAK frame** | Complete reconciliation artifact | The AE/AR frame quotes the offending field VALUE (PHI) — re-introduces exactly what #120 excludes from durable storage | **Rejected (→ scrubbed reason + code/phase only)** |
| **Plain `destination_name = ic.name` for ack rows** | Simplest | An inbound named identically to an outbound destination would corrupt that destination's per-destination authoritative-reply ordering | **Rejected (→ `\x1fack:` sentinel + `kind` seq predicate)** |
| **A new endpoint/permission for ACKs** | Explicit surface | The 0013 route already does the body/detail gating + audit; a second endpoint risks an inconsistent re-implementation | **Rejected (→ reuse `GET /messages/{id}/responses`)** |
| **(§7) Put pre-ingress events in `message_events` with a null `message_id`** | One table, reuses cipher/purge | **Fatal:** `message_events.message_id` is `NOT NULL REFERENCES messages(id)`; a placeholder `messages` row would inflate received-volume counts and pollute the finalizer | **Rejected (→ new `connection_event` table, §7.1)** |
