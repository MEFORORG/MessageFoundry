# ADR 0021 — Inbound ACK/NAK capture ("Response Sent"), an increment of ADR 0013

- **Status:** **Proposed.** Drafted, awaiting sign-off — no code yet. Conceptually "ADR 0013 Increment 3" (the inbound twin of 0013's outbound `response` capture), but it is a new decision and so carries its own number, **0021**. (`0019` and `0020` are taken — 0019 = the staged KeyProvider ADR; 0020 = protocol-diagnostic capture.)
- **Built:** nothing yet. This document is the design only.
- **Decision in one line:** record the ACK/NAK MessageFoundry **returns** to an inbound sender as a queryable, encrypted, audited, retention-purged artifact correlated to the inbound message — by **extending the existing ADR 0013 `response` table** with a `kind` discriminator plus two cheap plaintext typed columns (`ack_code`, `ack_phase`), reusing the entire 0013 cipher/purge/correlate/API/audit machinery, with capture done **synchronously** in `_handle_inbound` so there is no fire-and-forget vs key-rotation race.
- **Related:** [ADR 0013](0013-query-response-orchestration.md) (the `response` artifact table + `complete_with_response`/`correlate_response`/`purge_message_bodies` machinery this reuses), [ADR 0001](0001-staged-pipeline-architecture.md) (ACK-on-receipt + the listener path this hooks), [ADR 0004](0004-payload-agnostic-ingress.md) (non-HL7 inbounds return None / no ACK — nothing to capture), [CLAUDE.md](../../CLAUDE.md) §2/§9.

## Context

ADR 0013 captures the partner's reply to an **outbound** delivery. The symmetric gap is the **inbound** ACK/NAK MessageFoundry itself **generates and returns** to a sender — Corepoint's "Response Sent". `build_ack` builds the ACK string ([mllp.py:144-204](../../messagefoundry/transports/mllp.py)) and is called from **four** sites in `RegistryRunner._handle_inbound`: AR-decode ([wiring_runner.py:705](../../messagefoundry/pipeline/wiring_runner.py)), AR-parse ([735](../../messagefoundry/pipeline/wiring_runner.py)), AE-strict ([755](../../messagefoundry/pipeline/wiring_runner.py)), and AA-success ([776](../../messagefoundry/pipeline/wiring_runner.py)). `record_received`/`enqueue_ingress` already **return** the `message_id` ([store.py:1652, 1290](../../messagefoundry/store/store.py)) — but every site **discards** it. Nothing records the ACK we send; there is no store column, audit event, or test for an inbound ACK today (repo-wide grep for `record_ack`/`ack_sent`/`response_sent` = 0 hits). `_handle_inbound` is the **only** scope holding BOTH the store-returned `message_id` AND the `build_ack` string, and `build_ack` is already imported into the runner ([wiring_runner.py:70](../../messagefoundry/pipeline/wiring_runner.py)) — so capture is engine-side, with no `transports/` change and no `InboundHandler` contract change.

The sensitivity: the AA ACK echoes MSH (control fields — PHI), and an AE/AR NAK frame quotes the **offending field VALUE** from hl7apy's error string (the very PHI the existing #120 invariant excludes from durable storage — the code at [wiring_runner.py:750-758](../../messagefoundry/pipeline/wiring_runner.py) deliberately persists only `safe_text(joined)` and keeps the fuller AE frame transient).

## Decision

### §1 Extend `response`, do not add a new table

The `response` table is already immutable, finalizer-invisible (`_maybe_finalize_message` scans `FROM queue` only, verified [store.py:3583-3584](../../messagefoundry/store/store.py)), composite-PK cipher-passed (`_encrypt_existing_rows`/`reencrypt_to_active` over `body`+`detail`), and retention-purged on all three backends ([store.py:3342-3346](../../messagefoundry/store/store.py) + PG/SQL-Server twins). Reusing it (vs a new `inbound_ack` table) avoids ~2x per-backend DDL/cipher/purge duplication and **guarantees ACK PHI is encrypted at rest on SQL Server** (where `response.body`/`detail` are ciphertext — [sqlserver.py:1050-1051](../../messagefoundry/store/sqlserver.py) — but `message_events.detail` is plaintext).

### §2 Schema deltas (all three backends)

Three new columns on `response`: `kind` ('response' = existing outbound reply, the DEFAULT | 'ack_sent' = inbound ACK), `ack_code` ('AA'|'AE'|'AR'|'CA'|'CE'|'CR'; NULL for 'response'; non-PHI disposition metadata, plaintext), `ack_phase` ('decode'|'parse'|'strict'|'ingest'; NULL for 'response'; plaintext). Column reuse for an `ack_sent` row:
- `destination_name` = a **sentinel-prefixed synthetic name** `"\x1fack:" + ic.name`. The leading `\x1f` (unit separator, already the artifact_ref delimiter at [store.py:2152](../../messagefoundry/store/store.py)) cannot occur in a real connection name (names follow `[TYPE]_[PARTNER]_[MESSAGE]`, identifier-validated), so the ack row's `(message_id, destination_name)` key is **provably disjoint** from every outbound destination's key even if an operator names an inbound identically to an outbound.
- `response_seq` = `1 + COALESCE(MAX(response_seq),0) WHERE message_id=? AND destination_name=? AND kind='ack_sent'` — the **`kind` predicate is mandatory** in the seq SELECT. One ACK per received message in practice → seq=1. Append-only, replay-stable by the same argument as ADR 0013.
- `body` = **only for `ack_code='AA'`** (the success ACK echoes MSH but quotes no rejected field value) → `self._enc(...)`. For **every NAK** (AE/AR/CE/CR) `body = NULL`.
- `outcome` = 'accepted' (AA/CA) | 'rejected' (AE/CE/AR/CR).
- `detail` = a **mandatory `safe_text`-scrubbed**, bounded reason for a NAK (e.g. `safe_text(reason)[:200]`) → `self._enc(...)`; NULL for AA.
- `captured_at` = ACK generation time.

**PHI rationale for the AA-body/NAK-NULL split:** persisting an AE/AR frame would re-introduce exactly the PHI-quoting field value the #120 invariant excludes from durable storage. So for NAKs we persist only the scrubbed reason + code/phase — which IS the reconciliation artifact ("what code at 14:32, and why AR?"). The AA frame quotes no rejected value (it echoes MSH/MSA control fields), so storing it (encrypted) matches ADR 0013's outbound-body semantics.

Per-backend DDL + migration (no pre-existing `response` migration block — authored from scratch): **SQLite** `_SCHEMA` CREATE + a NEW `_migrate` guarded block (`PRAGMA table_info(response)` then `ALTER TABLE response ADD COLUMN` for the three; static literal names; existing rows backfill `kind='response'` via the DEFAULT). **Postgres** `_SCHEMA` CREATE + a NEW response branch inside `_migrate_lease_columns` (the information_schema-gated migrator — do NOT use a bare per-open `ADD COLUMN IF NOT EXISTS`, which takes ACCESS EXCLUSIVE every open). **SQL Server** `_SCHEMA` inline CREATE + guarded `IF COL_LENGTH('response','kind') IS NULL ALTER TABLE response ADD kind NVARCHAR(32) NOT NULL CONSTRAINT df_response_kind DEFAULT 'response'` (+ ack_code/ack_phase NULLable) — adding a NOT NULL column with a **constant** default is metadata-only (no rewrite) on SQL Server 2012+/2016+ (covers the CI 2022 container); **a migration test must time this ALTER on a pre-populated table**, with a documented fallback (NULLable add → backfill → set NOT NULL in batched steps) if any rewrite is observed. **NO `ix_response_ack` index in v1** on any backend (its only consumer — fleet ack-code analytics — is deferred; a filtered index is a SET-options foot-gun on the `response` INSERT hot path for zero v1 benefit). `CapturedResponse` ([store.py:133-145](../../messagefoundry/store/store.py)) gains `kind: str = 'response'`, `ack_code: str | None = None`, `ack_phase: str | None = None` (defaults keep outbound construction sites unaffected).

### §3 Capture hook (`_handle_inbound`, synchronous)

New store method `record_ack_sent(*, message_id, inbound_name, ack_body, ack_code, ack_phase, outcome, detail=None, now=None)` on the Store protocol + all 3 backends: synthetic `dest = "\x1fack:" + inbound_name`; under the backend's single-transaction discipline (the `complete_with_response` pattern — explicit BEGIN/single commit/rollback under `self._lock` on SQLite, `conn.transaction()` on PG, one cursor/commit with a leading SELECT to open the txn on SQL Server) it does `SELECT 1+COALESCE(MAX(response_seq),0) ... AND kind='ack_sent'` then `INSERT INTO response (... body=enc(ack_body), outcome, detail=enc(detail), ..., kind='ack_sent', ack_code, ack_phase)`. It touches **NO `queue` row**, calls **NO `_maybe_finalize_message`**, emits **NO `delivered` event**. SQL Server inlines the empty-guard exactly as `complete_with_response` ([sqlserver.py:1050-1051](../../messagefoundry/store/sqlserver.py)): `enc_body = self._cipher.encrypt(ack_body) if ack_body else ack_body` (there is no `self._enc` on that backend).

Wiring in `_handle_inbound` (un-discard the `message_id`, capture SYNCHRONOUSLY before returning the ACK): at each of the four ACK-emitting sites capture the returned id and, if `ic.capture_ack`, `await self._capture_ack(...)` before returning the ACK string. AR-decode → `ack_body=None, code='AR', phase='decode', detail=safe_text('decode error')`; AR-parse → `ack_body=None, code='AR', phase='parse', detail=safe_text(str(exc))[:200]`; AE-strict → widen `_record` ([wiring_runner.py:778](../../messagefoundry/pipeline/wiring_runner.py), single caller) to RETURN the id, then `ack_body=None, code='AE', phase='strict', detail=persisted[:200]` (the **already-scrubbed** `f"strict-validation failed: {safe_text(joined)}"`, NOT `joined` — the unscrubbed AE frame is NEVER persisted); AA-success → `ack_body=ack, code='AA', phase='ingest', outcome='accepted', detail=None`. No-ACK paths (`AckMode.NONE`; non-HL7 at [710-723](../../messagefoundry/pipeline/wiring_runner.py); File source) capture NOTHING (the table records only ACKs actually generated — exactly Corepoint's "Response Sent" semantics).

`_capture_ack` is **fail-soft on the store write**: the INSERT is `await`ed (transactionally adjacent — no scheduling-delay window vs rotation) but a store failure is **swallowed + logged WARNING** (`safe_exc`, never the body), so a capture error can never flip an AA to a NAK or tear down the MLLP per-connection loop ([mllp.py:615](../../messagefoundry/transports/mllp.py)). **Why synchronous (rotation-race fix):** a deferred `create_task` capture could land under the OLD key in a region `reencrypt_to_active` already scanned past, defeating rotation for that row. A synchronous, lock-serialized INSERT is naturally serialized against the rotation pass exactly like `complete_with_response` — written under whatever key is active and reachable by active-write + retired-key-retention. The `create_task`, inflight cap, and "best-effort" loss window are removed; the only loss window is a swallowed store-write failure (logged WARNING).

### §4 Config (per-inbound, OFF by default)

First-class field on `InboundConnection` ([wiring.py:1183-1203](../../messagefoundry/config/wiring.py)) — `capture_ack: bool = False` — NOT a transport-settings entry, because the RegistryRunner (pipeline) does the capture, so it belongs with `content_type`/`ack_after` (runner-honored fields). The ADR documents that this is **not** a structural mirror of `capture_response` (which lives in `spec.settings` on the outbound); it is the inbound analogue. Thread through the `inbound()` factory ([wiring.py:1453](../../messagefoundry/config/wiring.py)) and `build_inbound_connection` ([wiring.py:1353](../../messagefoundry/config/wiring.py), the shared code-first + connections.toml choke point): guards (raise `WiringError`, caught by `messagefoundry check`/dry-run) — `capture_ack=True` requires `ack_mode != AckMode.NONE` and `content_type is ContentType.HL7V2`. Add `capture_ack` to `_INBOUND_KEYS` and decode via `_require_bool(...)` defaulting False (a top-level connection-table key, not a `[settings]` key). No row-count cap needed (one row per received HL7 message, naturally bounded by inbound volume + retention).

**NEW inbound-side runner-start fail-closed guard.** The existing outbound guard ([wiring_runner.py:386-395](../../messagefoundry/pipeline/wiring_runner.py)) loops `registry.outbound` and keys on a destination attribute, so it structurally cannot see `ic.capture_ack`. Add a SEPARATE inbound loop at runner start (near [wiring_runner.py:404](../../messagefoundry/pipeline/wiring_runner.py)): for each `ic` with `capture_ack`, raise if `not supports_response_capture`, and re-validate the wiring guards (`ack_mode != NONE`, `content_type is HL7V2`) so a hand-edited `connections.toml` that bypassed `build_inbound_connection` still fails closed.

### §5 API + RBAC + audit (reuse the existing route)

Reuse `GET /messages/{message_id}/responses` ([app.py:1094-1143](../../messagefoundry/api/app.py)) — NO new endpoint, NO new permission. `correlate_response` ([store.py:2162-2184](../../messagefoundry/store/store.py)) now also returns `ack_sent` rows: add `kind, ack_code, ack_phase` to the SELECT projection ([store.py:2168](../../messagefoundry/store/store.py)) and the `CapturedResponse` construction ([store.py:2174](../../messagefoundry/store/store.py)). The `ORDER BY destination_name, response_seq` stays — ack rows sort under their `\x1fack:` synthetic name, distinct from every real destination, so the "latest seq per destination is the authoritative reply" semantics for outbound rows are **provably unaffected**. The route already gates `require_phi_read(Permission.MESSAGES_READ)`, 404s out-of-channel-scope with `_audit_channel_denied`, splits `body` on `MESSAGES_VIEW_RAW`, runs `redact_unauthorized` (nulls `detail` for callers lacking `MESSAGES_VIEW_SUMMARY`, [field_authz.py:67-68](../../messagefoundry/api/field_authz.py)), and emits `response.read` audit + `record_view` when bodies exposed — ack rows ride ALL of it unchanged. `CapturedResponseInfo` ([api/models.py:72-82](../../messagefoundry/api/models.py)) gains `kind: str`, `ack_code: str | None`, `ack_phase: str | None` — non-PHI disposition metadata, NO new `field_authz` entry (PHI `body` stays on the view_raw split; `detail` stays on the view_summary `field_authz` gate). The console "Response Sent" view consumes the same endpoint filtered to `kind=='ack_sent'`. Fleet ack-code analytics (`GET /responses?kind=ack_sent&ack_code=AR`) + its index are **deferred** to ship with their consumer.

### §6 PHI: encryption + retention (ZERO new cipher/purge code)

`ack_body` (AA only) → `self._enc(...)` into `response.body`; scrubbed `detail` → `self._enc(...)` into `response.detail` on SQLite+PG via the SAME `_enc` chokepoint; SQL Server inlines `self._cipher.encrypt(x) if x else x`. `kind`/`ack_code`/`ack_phase` are non-PHI plaintext. Cipher migration + rotation: `_encrypt_existing_rows` and `reencrypt_to_active` already iterate `response` body+detail by composite PK — ack rows are just more rows, ZERO change; the synchronous-capture decision keeps every ack row reachable by the rotation pass. Retention: `purge_message_bodies` already nulls `response.body`+`detail` in place for the eligible set ([store.py:3342-3346](../../messagefoundry/store/store.py) + PG/SQL-Server twins) — ack rows purge identically (NAK rows already have `body=NULL`); metadata (kind/ack_code/ack_phase/outcome/captured_at) survives for audit. **No RetentionPass change.**

## Consequences

**Positive**
- Corepoint's "Response Sent" becomes a queryable, encrypted, audited, retention-purged artifact correlated to the inbound message, reusing five subsystems of ADR 0013 machinery with **zero new cipher/purge code** and one new store method + three columns.
- `response` is not a queue stage; `_maybe_finalize_message` scans `FROM queue` only → an `ack_sent` row can NEVER pin a message out of `PROCESSED` or flip it to `ERROR`. Capture happens AFTER the ingress/record commit, so persist-before-ACK ordering is untouched.
- Synchronous capture eliminates the fire-and-forget vs key-rotation race entirely and removes the `create_task`/inflight-cap/best-effort-loss machinery.
- **Byte-identical when off** (`capture_ack=False`, the default): the four ACK sites keep discarding the id, no `response` row is written.
- NAK bodies are NOT persisted — the PHI-quoting AE/AR frame stays a transient wire ACK; only the scrubbed reason + code/phase are durable (#120 preserved).

**Negative / costs**
- Synchronous capture adds one local INSERT of ACK latency on the accept path per received HL7 message when `capture_ack` is on (within budget — the ingress commit already dominates; OFF by default).
- **Capture is single-shot at receipt, never replayed.** `_handle_inbound` runs exactly once per physical receipt; `reset_stale_inflight` recovers routed/outbound rows but never re-invokes the listener path. This is correct (the ACK is single-shot on the wire), but it means there is no re-run path that re-derives the ack row — a swallowed store-write failure is the only loss window. (ADR 0013's "on a re-run the reply re-derives" framing does NOT apply here; struck.)
- The seq/PK rule must be keyed on `kind` and ack rows use the `\x1fack:` sentinel `destination_name`, or an inbound name colliding with an outbound destination name could corrupt per-destination authoritative-reply ordering — a deliberate, tested guard.
- Captures the ACK we **GENERATED**, NOT a wire-confirmed drain (the socket write is at [mllp.py:605](../../messagefoundry/transports/mllp.py) after the handler returns) — matches ADR 0013's "what we produced" semantics; a reconciliation user must know captured ≠ delivered-on-wire.

## Alternatives considered

| Alternative | Why considered | Why rejected | Verdict |
|---|---|---|---|
| **Extend `response` with a `kind` discriminator, capture synchronously** *(chosen)* | Reuses all of ADR 0013's cipher/purge/correlate/API/audit; guarantees ACK PHI encrypted at rest on SQL Server | Three new columns on a hot table; a `_migrate` block per backend | **Adopted** |
| **A new `inbound_ack` table** | Clean separation | ~2x per-backend DDL/cipher/purge duplication; would have to re-prove SQL Server detail-column encryption | **Rejected** |
| **Fire-and-forget `create_task` capture** | Keeps the ACK latency budget untouched | A deferred INSERT can land under the OLD key past the rotation scan, defeating rotation for that PHI row; adds an inflight cap + loss window | **Rejected (→ synchronous)** |
| **Persist the full NAK frame** | Complete reconciliation artifact | The AE/AR frame quotes the offending field VALUE (PHI) — re-introduces exactly what #120 excludes from durable storage | **Rejected (→ scrubbed reason + code/phase only)** |
| **Plain `destination_name = ic.name` for ack rows** | Simplest | An inbound named identically to an outbound destination would corrupt that destination's per-destination authoritative-reply ordering | **Rejected (→ `\x1fack:` sentinel + `kind` seq predicate)** |
| **A new endpoint/permission for ACKs** | Explicit surface | The 0013 route already does the body/detail gating + audit; a second endpoint risks an inconsistent re-implementation | **Rejected (→ reuse `GET /messages/{id}/responses`)** |
