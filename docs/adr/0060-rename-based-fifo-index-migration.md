# ADR 0060 — Rename-based migration for the re-keyed FIFO covering indexes (land ADR 0059's re-key on upgraded DBs)

- Status: **Proposed**
- Date: 2026-06-30
- Relates to: [ADR 0059](0059-seq-only-fifo-ordering.md) (seq-only per-lane FIFO ordering; #673) — **completes 0059's index re-key on *upgraded* DBs**; [ADR 0001](0001-staged-pipeline-architecture.md) (staged pipeline / `queue` table); [ADR 0055](0055-group-committer.md) (group-committer); issue #285 (per-lane FIFO no-skip). Supersedes, for the two FIFO covering indexes, the same-name `CREATE INDEX IF NOT EXISTS` / `IF INDEXPROPERTY(...) IS NULL CREATE` guard that a re-keyed definition silently no-ops past on an upgraded DB.

## Context

ADR 0059 re-keyed the per-lane FIFO covering indexes to trail in `seq` so the seq-only claim (`ORDER BY seq`; SQLite `ORDER BY rowid`) is an index-ordered scan, not a sort:

- SQLite (`store.py`): `ix_queue_fifo_out/in ON queue(stage, {destination_name|channel_id}, status)` — `created_at` dropped; `rowid` is the intrinsic, unindexable order.
- SQL Server (`sqlserver.py`) / Postgres (`postgres.py`): `... ON queue(stage, {lane}, status, seq)` — `created_at` removed from the key, `seq` retained as the trailing column.

But 0059 **kept the same index names** and emitted them through *name-existence* guards (`CREATE INDEX IF NOT EXISTS` on SQLite/Postgres; `IF INDEXPROPERTY(OBJECT_ID('queue'),'ix_queue_fifo_*','IndexID') IS NULL CREATE` on SQL Server). A name-existence guard cannot tell the OLD `created_at`-trailing definition from the NEW `seq`-trailing one under the same name:

- **Fresh DB:** no such index exists → the guard fires → the new seq-trailing definition is created → the claim is index-served. Correct.
- **Upgraded DB** (created before B3): the identically-named OLD `created_at`-trailing index already exists → the guard no-ops → the seq-only claim runs against a mismatched, less-selective index (a sort / scan). **0059's throughput win never lands on upgraded DBs**, and the stale `created_at` column lingers in the FIFO index leaf.

Every real deployment upgrades in place (adopters re-pin the wheel; the store schema evolves via the `_migrate` / `_migrate_lease_columns` on-open path), so the upgraded-DB case is the common one, and the `queue` table is the hot path 0059 exists to speed up. There is **no schema-version sentinel** in the store (no `user_version`/`schema_version`) — DDL is guard-based and idempotent — so the discriminator has to be encoded in the index identity itself.

The changed-definition-same-name set is bounded to exactly **`ix_queue_fifo_in` and `ix_queue_fifo_out` on each of the three backends** (the #673 diff altered no other index definition; `ix_queue_ready` / `ix_queue_message` / `ix_queue_body_ref` / `ix_queue_lease` are byte-identical). `cancel_pending(top_only=True)` changed only a query `ORDER BY` tiebreak (to `next_attempt_at, seq`) and is served by the unchanged `ix_queue_ready(stage, status, next_attempt_at)` — it contributes **no** stranded index. (This corrects ADR 0059 line 40's "re-keyed to `next_attempt_at, seq`" phrasing: the *query* changed, not an index definition; there is no cancel-head index on any backend.)

## Decision

Adopt a **rename-based migration**. Give the seq-trailing FIFO indexes new, self-describing names so the definition is distinguishable by name alone, and run an idempotent, statement-timeout-exempt on-open migration that drops the old-named index (if present) and ensures the new-named one exists, on all three backends, reusing each backend's existing on-open migration path and HA schema lock.

| Old name (stranded `created_at`-trailing def) | New name (`seq`-trailing def) |
|---|---|
| `ix_queue_fifo_in`  | `ix_queue_fifo_in_seq`  |
| `ix_queue_fifo_out` | `ix_queue_fifo_out_seq` |

A rename is chosen over keeping the name + probing the stale column-set because name existence is then a **correct** discriminator: `DROP old IF EXISTS` + `CREATE new IF NOT EXISTS` (SQL Server: INDEXPROPERTY-guarded) is fully idempotent, fresh-safe (drop no-ops), and upgraded-safe (rebuild under the new name) with **no** per-backend index-column-inspection helper (which we do not have and would be fragile).

Per backend, the DROP and CREATE are placed so they land in **one atomic commit**, reusing that backend's established schema/migration frame and HA lock:

- **SQLite** (`store/store.py`): BOTH the `DROP INDEX IF EXISTS ix_queue_fifo_in/out` and the new-named `CREATE INDEX IF NOT EXISTS ix_queue_fifo_in_seq/_out_seq` live in the `_migrate` staticmethod (via `db.execute`), ordered DROP-old **then** CREATE-new, and are **removed from the `_SCHEMA` `executescript` blob** (a pointer comment is left in their place, mirroring how `ix_queue_body_ref` is documented + created in `_migrate`). **The swap is NOT transactional on SQLite** — Python's `sqlite3` auto-commits DDL in the legacy isolation mode aiosqlite uses, so each `DROP`/`CREATE` commits as it runs, *not* under `open()`'s later `db.commit()`. It does not need to be transactional: the FIFO index is **correctness-neutral** (the claim orders by `rowid` and names no index, ADR 0059), so a crash in the `DROP→CREATE` gap leaves a lane transiently unindexed (claims stay correct, just slower) and the next open's **idempotent** re-run (`DROP IF EXISTS` / `CREATE IF NOT EXISTS`) converges to the seq-trailing pair. The migration runs at open, before serving, so the transient gap is never observed by a live claim; `DROP`-old before `CREATE`-new keeps the on-disk FIFO index count from ever doubling. No applock exists (single-writer by construction + `busy_timeout=5000` + the process-wide `asyncio.Lock`); the migration is idempotent so a racing second opener is a no-op. (The two server backends run the same swap inside a **real** schema transaction and therefore *also* get atomicity — below.)
- **SQL Server** (`store/sqlserver.py`): the two FIFO entries in the `_SCHEMA` statement list become guarded DROP-old + CREATE-new pairs (`IF INDEXPROPERTY(...) IS NOT NULL DROP` then `IF INDEXPROPERTY(...) IS NULL CREATE`), executed by `_ensure_schema` **after** the transaction-scoped `sp_getapplock(_SCHEMA_LOCK)`, in one committed batch via the EF-6 `_cursor` CM (cursor closed before pool release). `ONLINE=ON` is **not** used (Enterprise-only). The per-statement `command_timeout` (default 30 s) is **suppressed for the schema batch** (`raw.timeout = 0`), because an offline `CREATE INDEX` over a large backlog that exceeds 30 s would raise, roll back the batch, and re-fail on every restart — a **startup crash-loop**.
- **Postgres** (`store/postgres.py`): the new-named `CREATE INDEX IF NOT EXISTS` stays in `_SCHEMA`; the `DROP INDEX IF EXISTS` of the old names goes in `_migrate_lease_columns`, beside the existing `DROP TABLE IF EXISTS lane_leases` retired-object precedent — both inside the single `_ensure_schema` transaction under `pg_advisory_xact_lock`, so CREATE-new (in the `_SCHEMA` loop) precedes DROP-old (in the migrate step) **in one atomic transaction**. Because the path is inside a transaction, `CREATE/DROP INDEX CONCURRENTLY` is **unavailable**; a plain `CREATE INDEX` (SHARE lock on `queue` for the build) is used, consistent with the store's no-CONCURRENTLY convention. The pool-level `command_timeout` (default 30 s) is **suppressed for the DDL** via `SET LOCAL statement_timeout = 0` issued inside the schema transaction, for the identical crash-loop reason (`SET LOCAL` auto-reverts at commit).

The migration runs at store open, **before** any listener/worker serves traffic.

## Mixed-version fleet + live-rejoin hazards (required mitigations, not footnotes)

Two operational hazards are inherent to a rolling upgrade and are mitigated **by process**, since the store has no version sentinel to refuse a stale opener:

1. **Mixed-version fleet.** `open_store`→`_ensure_schema` runs on **every** node **before** leadership is determined. If a NEW (B10) binary migrates a shared SQL Server / Postgres DB (old gone, new present) and an OLD (pre-B10) binary then opens the same DB, the OLD binary's schema-init re-creates the stale `created_at`-trailing index under its old name next to the new one. Both indexes then exist; every `queue` INSERT maintains TWO covering indexes (write amplification on the throughput-critical path), and the two versions fight — repeatedly creating/dropping the index on the live hot table under the schema lock. The applock **serializes** the two openers but does **not converge** them, because they disagree on the target definition. **Mitigation (required, documented):** treat the B10 store-schema bump as a **stop-the-world / full-fleet upgrade** on the shared-DB (SQL Server / Postgres) backends — do NOT roll it into a live mixed-version cluster; bring the fleet to a single version through the bump.
2. **Live-rejoin contention.** A passive/restarting node that rejoins while the leader is already serving runs the migration's real `DROP INDEX` + `CREATE INDEX` on the **live** `queue` on its first-upgrade open. The schema applock serializes node-vs-node schema-init but does **not** serialize the migrating node's Sch-M (SQL Server) / SHARE (Postgres) lock on `queue` against the **leader's live claim workers**, which never take the applock. The rebuild can block live claims (or the leader's claims block the join's DDL until the timeout). This is a **liveness/availability** stall, not a wrong-result bug. **Mitigation (required, documented):** on the shared-DB backends, perform the version-bump restart during a **drain / maintenance window**, ideally single-node (or on the passive node first, before it is leader). The out-of-transaction Postgres `CREATE INDEX CONCURRENTLY` escape hatch (Residual risks) is the only non-blocking option and is the sole recourse for an adopter who must rejoin into a live HA cluster on the upgrade boot; SQL Server has no non-Enterprise online equivalent, so SQL Server HA **must** upgrade under drain.

## Why FIFO correctness is unchanged

The per-lane claim orders by `seq`/`rowid` alone (ADR 0059) and **names no index** — SQLite `ORDER BY rowid`, SQL Server `WITH (UPDLOCK, ROWLOCK) ... ORDER BY seq` with **no** READPAST, Postgres `ORDER BY seq ... FOR UPDATE` with **no** SKIP LOCKED of the true head. The planner picks the covering index, so an index affects only scan **speed**, never the result set or its order — the OLD index, the NEW `_seq` index, or NO index all return the identical head and identical contiguous-due prefix. The #285 no-skip lock semantics are the claim's lock hints + the SQLite global `self._lock`, none of which this migration touches. Renaming/rebuilding the covering index therefore cannot change which row is the head, at-least-once, the poison-guard (ADR 0055), the finalizer's sole disposition authority (its scans are `GROUP BY` with no `ORDER BY` — order-independent), or ACK-on-receipt. This migration is a **pure speed restoration** on upgraded DBs.

## Consequences

- Upgraded DBs adopt the seq-trailing FIFO index (0059's throughput win finally lands there).
- The stale `created_at` column leaves the FIFO index leaf on upgraded DBs (smaller, more selective index).
- First index in the store layer with a versioned (`_seq`-suffixed) name and the first `DROP INDEX` in the store layer (precedent: Postgres `DROP TABLE IF EXISTS lane_leases`).
- The B10 first-upgrade open is the **one** open that executes real DDL on the hot `queue` (a DROP + a rebuild), unlike every existing schema statement which is guarded to no-op on an already-migrated DB; re-opens after B10 are lock-free.
- Forward-only: a downgrade re-creates the old-named `created_at` index via the old build's guards (redundant but harmless coverage).

## Reliability-invariant checklist

- [x] **At-least-once / crash re-run idempotent:** migration is `DROP IF EXISTS` + `CREATE IF NOT EXISTS` (SQL Server: INDEXPROPERTY-guarded), safe to re-run. On the server backends the DROP+CREATE run inside the schema transaction, so a crash rolls the swap back atomically; on SQLite the swap is not transactional (sqlite3 auto-commits DDL) but is correctness-neutral + idempotent, so a crash in the gap leaves a lane transiently unindexed (claims still correct) and the next open converges. No queue rows touched on any backend.
- [x] **Strict per-lane / order-group FIFO (#285, no READPAST / no SKIP LOCKED of the true head):** ordering is seq-only and index-independent; the claim names no index and its lock hints are untouched. The migration changes index SPEED, never claim results.
- [x] **Poison-guard (ADR 0055):** attempts/enqueue logic untouched. Unchanged.
- [x] **Finalizer = sole disposition authority:** finalizer scans are `GROUP BY` with no `ORDER BY` (order-independent) and do not use the FIFO index. Unchanged.
- [x] **Count-and-log / ACK-on-receipt:** ingress row still committed before ACK; migration runs at open, before serving. Unchanged.
- [x] **HA two-node concurrent open (schema-init race):** serialized by the existing schema lock — `sp_getapplock('mefor:schema_init')` (SQL Server), `pg_advisory_xact_lock(2, 'mefor_schema_init')` (Postgres); SQLite is single-writer (busy_timeout + process lock) and the migration is idempotent. NOTE: this serializes schema-init vs schema-init; it does NOT serialize a migrating node's queue-rebuild lock against a LEADER's live claim workers — see the live-rejoin mitigation (drain-window upgrade).
- [x] **EF-6 SQL Server no-MARS cursor-close-before-release:** the SQL Server DDL runs on the `_ensure_schema` `_acquire()`/`_cursor()` path, so the cursor closes before the connection returns to the aioodbc pool. Pinned by the schema-init contract test.
- [x] **No unbounded lockout / no startup crash-loop:** runs once, at open, before serving; bounded by live queue depth (normally drained). The per-statement timeout is EXEMPTED for the schema DDL on both server backends (SQL Server `raw.timeout=0`; Postgres `SET LOCAL statement_timeout=0`) so a large-backlog rebuild cannot be killed → rolled back → re-crash on the next restart.

## Residual risks

- **Large-backlog upgrade (blocking rebuild).** An upgrade with a big pending `queue` pays a bounded blocking index rebuild at open: SQLite single-writer lock (no online build); SQL Server offline Sch-M under the applock; Postgres plain-CREATE SHARE lock (blocks writes) because the txn frame precludes CONCURRENTLY. Acceptable (once-per-DB, at cold start, on a normally-drained table). The statement-timeout exemption ensures this completes rather than crash-looping.
- **Postgres CONCURRENTLY escape hatch (out of the baseline; required only for live-HA-rejoin).** Moving the Postgres FIFO rebuild out of the schema transaction to `DROP INDEX CONCURRENTLY` + `CREATE INDEX CONCURRENTLY` is the only non-blocking option, but it MUST: run in its **own** implicit transaction (not the schema txn); hold a **session-scoped** `pg_advisory_lock` (NOT the txn-scoped `pg_advisory_xact_lock`); run with `statement_timeout=0`; and **detect + drop an INVALID index** (`pg_index.indisvalid=false`) left by a failed prior CONCURRENTLY build before re-creating. Kept out of the B10 baseline.
- **SQLite long rebuild vs `busy_timeout`.** On a very large populated DB the rebuild could exceed the 5 s `busy_timeout` for a *concurrent second opener* (a console/harness process, a second serve attempt) → a transient, non-corrupting, idempotent open failure. Changelog caveat included.
- **Downgrade re-creates the old index** via the old build's name-existence guard (redundant coverage; release notes: drop `ix_queue_fifo_in/out` manually if downgrading permanently).
- **Planner adoption verified at runtime, not in driver-free unit tests:** that each backend's planner actually picks the new seq-trailing index needs a runtime `EXPLAIN` (SQLite `EXPLAIN QUERY PLAN`; gated SQL Server / Postgres legs). The driver-free unit tests assert the DDL contract (new index present with the seq-trailing key / old gone), not the plan.
