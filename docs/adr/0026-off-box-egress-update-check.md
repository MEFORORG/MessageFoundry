## Context

#30 asks for an automatic "newer MessageFoundry version exists" signal surfaced in the console and the VS Code IDE, so an operator sees drift without hand-diffing `pyproject.toml` / `requirements.lock` against PyPI ([BACKLOG #30](../BACKLOG.md)). The feature has a real tension at its heart, and the backlog entry already names it as "the main design constraint": **a version check is an outbound network call, which conflicts with the on-premises-by-default, no-egress posture** ([CLAUDE.md](../../CLAUDE.md) §9; [SECURITY.md](../SECURITY.md) — the API binds `127.0.0.1` by default and no PHI leaves the local environment without explicit, reviewed configuration).

Why on-prem HL7 integration engines distrust "phone-home":

- **The deployment is a clinical PHI hub, often air-gapped.** A healthcare integration engine sits between EHRs and partners; many sites run it on an isolated VLAN with no general internet egress by policy. A background process that silently dials an external host (even one carrying only a version string) is, to a hospital security team, an **unaudited outbound channel out of a PHI system** — exactly the thing a CISO's egress controls exist to forbid. The reliability invariant deliberately avoids an external broker for the same reason; the engine is meant to be a self-contained, inspectable box.
- **It is a new egress surface for zero functional benefit to message flow.** Unlike a REST/FHIR destination (which *must* egress to do its job, and is therefore gated by the fail-closed `[egress].allowed_http` allowlist — [ADR 0022](0022-fhir-resource-codec-rest-client.md) §3.4), a version check moves no messages. An always-on phone-home would be a standing SSRF/DNS-exfiltration/supply-chain-tracking concern that buys the data plane nothing.
- **Dep staleness is already covered.** Deps are pinned in the hash-locked `requirements.lock`, and CI's **DEP-1** audit already flags vulnerable/stale dependencies out-of-band. A second, *runtime* dependency-scan mechanism inside the engine would duplicate that with a new always-on egress path — net new attack surface for a check the CI gate already performs (this is the §G trim, below).
- **The engine already knows enough to be useful with no network.** The running version is single-sourced as `__version__` ([messagefoundry/__init__.py:61](../../messagefoundry/__init__.py), Workstream F) and the intended/locked version is in the bundled `requirements.lock` / the installed distribution metadata. A "what I am running vs what this build pinned" diff needs **zero** outbound traffic.

The right precedent for *how* to bound an egress-capable feature already exists: the `[ai]` policy ([AI.md](../AI.md)) ships an AI assistant on an **OFF→PHI-safe spectrum**, **environment-clamped** so a central operator decision (off, or a cap on what may be sent) is honored on every workstation, with the most permissive states **unreachable by default** and clamped per `dev`/`staging`/`prod` posture. An off-box version check is structurally the same problem — an optional capability that crosses a trust boundary — so it should inherit that model rather than invent a new one.

## Decision

### §1 The MVP is a no-network "pinned-vs-current lock diff" (default + only build)

The **default and the only thing built in the MVP** is a passive, **zero-egress** comparison:

- **Inputs, both local:** the running version (`__version__`, [messagefoundry/__init__.py:61](../../messagefoundry/__init__.py)) and the **pinned/bundled** version recorded in the shipped `requirements.lock` (or the installed `messagefoundry` distribution metadata via stdlib `importlib.metadata`). No PyPI call, no DNS lookup, no socket — the comparison reads files/metadata already present in the install.
- **What it computes:** a small, non-PHI result — `current_version`, `pinned_version`, and a derived `update_available: bool` (current < pinned, i.e. the running process is older than what this build/lock expects). It carries **no message content, no dependency list, no host data** — only version strings.
- **The runner** clones the established background-monitor pattern of [cert_expiry.py](../../messagefoundry/pipeline/cert_expiry.py) / [retention.py](../../messagefoundry/pipeline/retention.py): a small, engine-owned task (started in `Engine.start`, awaited/cancelled in `Engine.stop`), **cooperatively cancellable** via an `asyncio.Event` stop signal, with one isolated pass per interval whose failure is logged and never takes the engine down (`run_once` is the deterministic, injected-clock test seam; the loop only governs cadence). Because the MVP does **no** I/O that could block, even the `asyncio.to_thread` hop is unnecessary for the diff itself; it is reserved for the future live path (§2).
- **It is engine-side and dependency-light** (stdlib + the existing alert/version seams only — like `cert_expiry`/`retention`), so it never pulls `api`/`console` into the engine and adds **no new dependency** ([CLAUDE.md](../../CLAUDE.md) §7).

This delivers #30's operator value — "you're running an older build than this install pins" — with **zero** of the egress concerns above.

### §2 The constrained future-egress envelope (DEFINED here, NOT built now)

A **live** check (compare against PyPI or a configurable internal index) is **explicitly not built in this slice.** This ADR records the **constrained envelope it must live inside** if and when it is built, so the boundaries are settled before any code is written and no future author re-litigates them:

1. **Off by default, env-clamped on the `[ai]` precedent.** The live check is an opt-in capability on an **OFF→on spectrum**, clamped by the instance's production posture exactly as `[ai].data_scope` is ([AI.md](../AI.md), `resolve_effective_policy`). A custom/unresolved environment clamps to the **strictest** setting (OFF) — an un-tuned install never silently dials out. Like the central AI *off*, the operator (not a developer or a default) decides.
2. **No new dependency — stdlib `urllib` only.** The HTTP call reuses the standard library, **never** `requests`/`httpx` ([CLAUDE.md](../../CLAUDE.md) §7; the same stdlib-only stance `WebhookTransport`/`send_plain_email` take in [alert_sinks.py](../../messagefoundry/pipeline/alert_sinks.py)).
3. **Constrained egress envelope mirroring `WebhookTransport`** ([alert_sinks.py:151-206](../../messagefoundry/pipeline/alert_sinks.py)): **https-only** (a plaintext `http://` index refused unless the explicit `MEFOR_ALLOW_INSECURE_TLS` dev escape is set — `insecure_tls_allowed()`); **no redirects followed** (reuse the `_NoRedirectHandler`/`_NO_REDIRECT_OPENER` so a 3xx cannot divert the probe to an internal host — ASVS 15.3.2 / SSRF); a **host allowlist** (an `update_index_allowed_hosts` analog of `webhook_allowed_hosts`, empty = the configured index only) so the check can be pointed at an **internal mirror** for air-gapped sites and cannot be coerced elsewhere; and the call runs **off the event loop** via `asyncio.to_thread` (blocking I/O, the `WebhookTransport.send` pattern).
4. **Advisory-only — never mutates anything.** It **never** auto-installs, **never** writes/edits `pyproject.toml` or `requirements.lock`, and introduces **no** auto-install/lock-mutating path. It only *reports* that a newer version exists. (This is a hard line, not a default — there is no "auto-update" knob to flip.)
5. **RBAC-gated + audited on any outbound check.** Any *outbound* check is gated by an existing permission and writes an `audit_log` row (**never** message content — the `RetentionRunner._audit` pattern, [retention.py:255-272](../../messagefoundry/pipeline/retention.py), which audits `actor="system"` for an autonomous background pass; an *operator-initiated* check instead carries the requesting operator's actor + the index host + result), so an egress event out of a PHI system is always attributable and reviewable ([SECURITY.md](../SECURITY.md) deny-by-default + audit posture).

Defining the envelope now (without building it) means the future live check is a bounded, pre-reviewed increment — not a fresh security debate — and it cannot regress the loopback/no-egress default.

### §3 Config shape — a new `[update_check]` section, off by default

A new `[update_check]` `_Section` in [config/settings.py](../../messagefoundry/config/settings.py), placed beside the other engine-monitor sections (`CertMonitorSettings`/`RetentionSettings`), with the same **off-by-default / `enabled` property gates `start()`** discipline those runners use ([cert_expiry.py:117-130](../../messagefoundry/pipeline/cert_expiry.py), [retention.py:99-122](../../messagefoundry/pipeline/retention.py)):

| Key | Type | Default | Notes |
|---|---|---|---|
| `enabled` | bool | `true` | The **no-network local diff** is cheap and PHI-safe, so the passive §1 check is on by default; set `false` to suppress the `/status` field + the alert entirely. |
| `check_interval_seconds` | float | `86400.0` | Diff cadence (the diff is trivial; daily is ample). The loop wakes immediately on stop, like `cert_expiry._sleep`. |
| `mode` | enum | `"local"` | **`"local"` (no-network diff, the only MVP value)** · `"live"` is **defined but rejected at load until the §2 envelope is built** — a `WiringError`/`ValueError` "live update-check is not implemented" so the value can't silently become a phone-home. |
| `index_url` | str | — | **Future (§2 live mode only).** The package index / internal-mirror base URL; https-only, env-clampable. Unused in the MVP. |
| `index_allowed_hosts` | list[str] | `[]` | **Future (§2).** Host allowlist for the live probe (`webhook_allowed_hosts` analog). Empty = the `index_url` host only. Unused in the MVP. |

The MVP ships `enabled` + `check_interval_seconds` + `mode` (clamped to `"local"`); the `index_*` keys are documented as **forward-compat, accepted-but-unused** (the same "parses so a forward-looking config loads, but nothing in this build reads them" stance `AiSettings` takes for its broker keys, [settings.py:809-815](../../messagefoundry/config/settings.py)). The §2 env-clamp (a posture-derived ceiling that forces `mode="live"` back to `"local"`/OFF outside an explicitly-permitted posture) lands **with** the live mode, not in the MVP — but the section is shaped so adding it is additive.

### §4 `/status` + AlertSink integration — riding #21 and #5, changed in lockstep

The result surfaces through **two existing, authoritative engine signals** — no bespoke channel, honoring #30's "the engine emits one signal the console and IDE both render" shape and the one-way dependency rule (the thin UIs consume the API; they never make their own PyPI call):

1. **One additive `/status` field (on the pre-existing `GET /status`, after the #21 observability wave).** `GET /status` already returns `SystemStatus`/`EngineInfo` ([api/app.py:1311-1343](../../messagefoundry/api/app.py), [api/models.py:255-280](../../messagefoundry/api/models.py)) gated by `Permission.MONITORING_READ`. Add **one additive, optional field** — an `update: UpdateInfo | None` (or `update_available`/`current_version`/`pinned_version` on `EngineInfo`) carrying the §1 result (version strings + the bool, **no PHI**). It is additive and `None` when `[update_check]` is disabled, so the existing payload is unchanged when off.
2. **Optionally one `update_available` AlertSink event (rides the #5 alert framework).** When the diff finds an update, the runner may emit a **single new advisory event** through the existing `AlertSink`, exactly as `RetentionRunner`/`CertExpiryRunner` emit `storage_threshold`/`cert_expiry`. Adding the event type means changing the four AlertSink touch-points **in lockstep** (the plan calls this out explicitly):
   - `AlertSink` Protocol — a new `update_available(...)` method ([alerts.py:27-56](../../messagefoundry/pipeline/alerts.py));
   - `LoggingAlertSink.update_available(...)` — log at `WARNING`, version strings only ([alerts.py:59-100](../../messagefoundry/pipeline/alerts.py));
   - `NotifierAlertSink.update_available(...)` — enqueue the event (the `cert_expiry`/`storage_threshold` `_emit` pattern; `connection` keyed to a stable label like the package name so the re-alert throttle works) ([alert_sinks.py:368-404](../../messagefoundry/pipeline/alert_sinks.py));
   - `_ALERT_EVENT_TYPES` in [config/settings.py:932-934](../../messagefoundry/config/settings.py) — add `"update_available"` so an `[alerts].rules` `event_type` may match it (the `frozenset` and the `AlertSink` methods must mirror each other, by the comment there).
   - (`_subject` in [alert_sinks.py:119-121](../../messagefoundry/pipeline/alert_sinks.py) needs no change — it reads `event["type"]`/`["connection"]`/`["severity"]` generically — but the lockstep set is named so the build re-verifies it; the new event carries **only** version strings, so the webhook JSON / email subject stay PHI-free, like every other alert.)

The event is **best-effort and advisory** (the sink contract — never raises, never blocks delivery); a missing alert never affects message flow, and the `/status` field is the authoritative signal.

### §5 The §G trim — the auto dependency-vulnerability-scan half is NOT built

#30 originally bundled an auto **dependency**-vulnerability/update scan alongside the MEFOR-version check. That half is **dropped** ([MULTISESSION-PLAN-3.md](../releases/MULTISESSION-PLAN-3.md) §G; [BACKLOG #30](../BACKLOG.md)): CI's **DEP-1** audit + the **hash-locked `requirements.lock`** already cover dependency staleness/vulnerability out-of-band. Building a **second, runtime** dependency-scan mechanism inside the engine would (a) duplicate the CI gate, (b) require its own always-on egress to a vulnerability index — the exact phone-home this ADR is constraining — and (c) tempt an auto-install/lock-mutating path this ADR forbids (§2.4). So the MVP keeps **only** the off-by-default (prefer no-network) **MEFOR-version** advisory. Dependency posture stays where it belongs: the lock + DEP-1.

## Consequences

**Positive**

- **The on-prem / air-gapped / loopback-default posture is preserved by default.** The MVP makes **zero** outbound calls; an air-gapped site gets the version-drift signal with no egress whatsoever, and nothing in the default install dials out ([CLAUDE.md](../../CLAUDE.md) §9, [SECURITY.md](../SECURITY.md)).
- **No new dependency, no new always-on egress surface.** The runner is stdlib-only and clones two already-shipped, reviewed background runners; the future live path reuses `WebhookTransport`'s hardened envelope rather than inventing one.
- **One authoritative signal, thin UIs.** The console (PySide6, over the API) and IDE banner both consume the same `/status` field / `update_available` alert — never their own PyPI call — honoring the engine-emits / one-way-dependency rule. The thin `console/status.py`+`client.py` feed + `ide/src/home.ts` banner are handed to Lane D (§F).
- **The hard tension is settled, not re-litigated.** The constrained future-egress envelope is written down (env-clamped, https-only/no-redirect/host-allowlist, stdlib-only, advisory-only, RBAC+audited), so a future live check is a bounded increment, not a fresh security debate — and it structurally cannot regress the no-egress default.
- **No duplicated mechanism.** Dependency staleness stays on CI DEP-1 + the hash-locked lock; the engine doesn't grow a second runtime scanner (§5).
- **Byte-identical when off** (`[update_check].enabled = false`): no runner task, no `/status` field, no alert — the existing payload and behavior are unchanged.

**Negative / costs**

- **The local diff only detects "older than this build pinned," not "newer than this exists upstream."** A site running the latest *pinned* build gets `update_available = false` even if a newer `messagefoundry` release exists on PyPI — that genuinely-upstream signal needs the §2 live mode, which is deliberately deferred. (Operators audit upstream releases out-of-band today, exactly as BACKLOG #30's "why deferred" notes.)
- **A new AlertSink event type adds a lockstep-change obligation.** `update_available` must be added to the Protocol + both sinks + `_ALERT_EVENT_TYPES` together (and the build must re-verify `_subject`/rule-matching), or an `[alerts].rules` `event_type = "update_available"` would fail validation — a deliberate, tested coupling (§4).
- **A small additive `/status` field + a new `[update_check]` section.** Minor surface growth on two hotspots; both additive and `None`/off by default.
- **#30 carries real cross-file contention.** It touches `config/settings.py`, `pipeline/engine.py`, `api/app.py`, `api/models.py`, `pyproject.toml` (version/section), plus `console/*` and `ide/` — which is precisely why it gets **no separate worktree** and joins a live owning lane on trigger (§F), accepting some sequencing friction in exchange for no merge-race.

## Alternatives considered

| Alternative | Why considered | Why rejected | Verdict |
|---|---|---|---|
| **No-network "pinned-vs-current lock diff"** *(chosen MVP)* | Delivers #30's operator value with zero egress; reuses local `__version__` + the bundled lock; clones the `cert_expiry`/`retention` runner | — | **Adopted (default + only MVP build)** |
| **Build a live-egress PyPI/index check now** | The fullest signal ("a newer release exists upstream") | A standing outbound call out of a PHI system contradicts the loopback/no-egress default ([CLAUDE.md](../../CLAUDE.md) §9); buys the data plane nothing; better delivered as a bounded, env-clamped opt-in once its envelope (§2) is built and reviewed | **Rejected now → defined as the constrained §2 future envelope (off by default)** |
| **Auto dependency-vulnerability scan (the other #30 half)** | "One place to see stale/vulnerable deps + a new MEFOR version" | CI **DEP-1** + the hash-locked `requirements.lock` already cover dep staleness/vulnerability; a runtime scanner duplicates that, needs its own always-on egress, and tempts an auto-install/lock-mutating path this ADR forbids | **Rejected (§G trim)** |
| **A second runtime mechanism / bespoke phone-home channel** (a dedicated endpoint or its own HTTP client, not riding #21/#5) | Decouples the feature from the alert/metrics surfaces | Reinvents the `/status` + `AlertSink` signals; risks a divergent, un-hardened egress client (no shared no-redirect/allowlist/stdlib posture); the thin UIs would be tempted to call out themselves, breaking the one-way rule | **Rejected (ride #21 `/status` + #5 AlertSink)** |
| **Auto-install / auto-update on detecting a newer version** | "Self-updating engine" | Mutating a running clinical PHI hub's deps/lock unattended is unacceptable; violates the advisory-only + never-mutate-the-lock guardrail (§2.4) | **Rejected (advisory-only, never auto-install / never mutate the lock)** |
