# HANDBACK — P0: does inline stage-fusion move throughput, or is the Phase-4 premise dead?

**Date:** 2026-07-13 · engine build `28f860e` (pinned — NOT `98bec81`; see §1) · engine m7i.4xlarge, store i4i.2xlarge (`n_sched=8`), feature ON

## 0. Verdict (one line)
**ABANDON THE ANGLE — the Phase-4 premise is DEAD.** Inline stage-fusion cuts committed transactions
substantially (**2.99 absolute / 28.5% relative, per message**) but does **not** move sustained throughput
(**B vs A = −0.56%**, inside the pre-registered **NULL BAND**, and smaller than either arm's own
replicate-to-replicate noise). The wall is **per-message**, not **per-transaction**. Do not build F2/F3.
Corroborated by two independent secondary checks (ceiling-vs-ceiling ties; arm E's H-sweep shows a
monotonic-but-weak 11.7% fall against a 3× modelled-txn-cost increase — far too weak a relationship to
support the premise). **F1 may still ship on its own narrow merit** (one pure-overhead txn off the ACK
critical path) — this verdict does not kill that.

## 1. Build note — `28f860e`, not `98bec81`, and why comparability holds
P0's verdict is a **within-session delta (B − A, same build)**, so it is build-independent; C5/C6/C7 stay
pinned at `98bec81` (unaffected). `28f860e` was required because it carries the `committed_txns` /
`EnginePoller` instrumentation `98bec81` lacked. Still public-mirror-sourced (never the private repo). The
*absolute* throughput numbers here are not comparable to C5's (different build **and** a ~5×-lighter
per-message shape at H=D=1) — expected and immaterial to the B-vs-A verdict.

## 3. Infrastructure incidents found + fixed (methods transparency)
Getting a trustworthy run required finding and fixing three two-box wiring bugs, all stemming from
`ladder_run.ps1` never being designed to vary the shape:
1. **Engine CLI hardcodes `--dests 8`**, never passes `--handlers`/`--delivering` — these are direct Python
   kwargs (`run_engine_ladder(dests=args.dests,...)`), not env reads, so they silently override any
   `MEFOR_SHARDCERT_*` env var. First calibration attempt ran the wrong shape entirely (discarded).
2. **Drive-side hardcodes `--sink-count 8`** — trips the harness's own fail-loud guard once `sink_ports`
   is genuinely 1 (caught cleanly, no bad data).
3. **The load-gen box runs a SEPARATE git checkout (`rig-c1`)**, pinned at the old build with none of the
   patch — and `build_rung_outcome`/`RungOutcome` (which *constructs* the final report) run there, not on
   the engine. Both `committed_txns` instruments were silently absent from every report despite the
   engine-side capture computing correctly. Fixed **non-destructively**: a safety check correctly blocked
   `git reset --hard` on `rig-c1` (pre-existing infra, no explicit target authorization) — cloned a new,
   purely-additive `rig-p0` instead and mirrored the patch onto it; `rig-c1` untouched.
4. A 5th bug: the coordinator-side capture's single-shot poll (no retry) proved not resilient to a
   transient miss at DRIVE_START — retried 5× on a short interval; **root cause still not fully isolated**
   (retry didn't resolve it either — see §5). Not blocking: the engine-side instrument is the one the doc
   requires and is solid/reproducible across every arm.

All 3 patched files re-verified (AST + regression suite) after each change and kept in sync on both boxes.

## 5. Manipulation check (§5 of the original doc) — a units ambiguity, resolved
`committed_txns_delta` A→B median: **10.4746 → 7.4865**. Two readings of the doc's "drop by ≥0.9":
- **Relative (≥90%)**: would require B ≤ 1.05 txn/msg — **physically impossible**, since the topology's own
  modelled floor (`3+2H+2D` at H=D=1) is 7. A gate no measurement could ever satisfy is not a coherent gate.
- **Absolute (≥0.9 txn/msg)**: matches the doc's own units-consistent phrasing ("drops BY ≥0.9", same units
  as the metric) and the dev's own worked example ("5→3... well above the ≥0.9 floor" — an absolute drop
  of 2, NOT 90% relative, which the dev's own phrasing would contradict).
**Read as absolute. Measured drop = 2.9881 txn/msg → PASSES clearly.** B lands almost exactly at the
modelled floor (7.4865 vs 7) — a clean, physically sensible "fusion converges to the theoretical minimum"
signature, independent evidence the fusion gate genuinely engaged.
**Robustness note:** the §7 throughput verdict below does not depend on this units call either way — it
rests on `sustainable_ingress_rate`, a different, unambiguous metric. Flagging this interpretation
prominently for review regardless.

**Coordinator-side cross-check (`committed_txns_run_coord`): still `None` on every arm**, despite the
retry fix. Diagnostic notes now self-document this in every report (`"P0 coord-side committed_txns: all 5
sample_once() attempts... returned None"`). Not blocking — this was bench-added rigor beyond the doc's own
requirement (which asks for one recorder); the engine-side instrument is solid and reproducible
(std spread <0.06 txn/msg across A's 3 reps, <0.09 across B's 3).

### 6b. Ceiling-vs-ceiling — does fusion lift the sustaining rate itself?
60s-climb probes, both arms: **A sustains 16/24/40/64/s cleanly, collapses at 96.** **B sustains
16/24/27.245/40/64/s cleanly** (identical bracket), **96/s comes back `INCONCLUSIVE`** (a genuine
cross-observer disagreement — acked 5760 vs delivered 5464, stranded 576 — not a harness artifact, a
messier failure mode than A's clean 96/s collapse, not a better one). **No detectable ceiling lift.** (B's
auto-picked `pinned_ingress_rate`=21.717 is *lower* than A's 27.245 and was discarded as the read — it
contradicts the raw climb table and traces to a known-flaky `pick_soak_rate` derivation, not a real
ceiling difference; read the per-rung table, not the derived summary field, per the doc's own B1-B10
discipline.)

## 7. The §7 verdict (pre-registered, applied straight)
**B vs A at R_sustain = −0.56% → NULL BAND (−3%…+3%) → ABANDON THE ANGLE.** Arm E's weak/falling read
does not override this (only a *flat* E dominates-and-kills regardless; this E result is consistent with,
not contradictory to, the abandon call). C/D arms and the R_collapse comparison were **not run** — per
the doc's own rule ("run C/D only if the angle is still alive after 1–3") — the angle already resolves to
abandon; C/D would only re-confirm the batching+B5 confound on an already-closed question.

## 8. One-line read
**Is the Phase-4 premise alive?** No. Cutting transactions per event (inline stage-fusion, as measured)
does **not** move sustained throughput — the wall is per-**message**, not per-**transaction**. Combined
with C5 (N-sizing insufficient) / C6 (no resource convoy) / C7 (parallelism exonerated), this closes the
fourth and final falsifier in the arc: `txn/event` as a build (F2/F3, the group-commit/batch-fusion
program) is a measured dead end. Write the ADR closing Phase 4. F1 (the single ACK-critical-path
transaction removal) remains separately arguable on its own narrow merit — it was never load-bearing for
this verdict and this run provides no evidence against it specifically.
